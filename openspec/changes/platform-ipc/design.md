# Design — Platform IPC transport

Records the technical decisions behind the `platform-ipc-transport` capability. Part of the macOS-first multiplatform port (vault: `Projects/nergal/Multiplatform port scoping.md`, cluster A1/A2). macOS is the first target; Windows is a later iteration.

## Context

All of Nergal's IPC runs over Linux primitives:

- **Hook event socket** (`hooks/server.rs`): `tokio::net::UnixListener::bind` (`:203`), fire-and-forget newline-delimited messages, `set_permissions(0o600)` in a bare (ungated) block (`:206-209`). The hook CLI (`hooks/cli.rs:3`) imports `std::os::unix::net::UnixStream` and connects at `:16,47,178,250,292`.
- **MCP daemon** (`mcp/transport.rs`): `UnixSocketTransport` already wraps `UnixListener`/`UnixStream` with length-framed (4-byte LE) messages. Its perms (`:83`) and `peer_uid` via `SO_PEERCRED` (`:113-124`) are **already `#[cfg(unix)]`-gated**, and the module doc already states the intent that "a future Windows named-pipe transport drops in without touching dispatch." This is the reference pattern for the rest of the surface.
- **Blocking request/response** uses POSIX FIFOs: plan-review creates `/tmp/nergal-plan-{pid}.fifo` via `mkfifo` (`hooks/cli.rs:142,146`) and blocks on `read_to_string`; ask-user uses `nergal-ask-*` FIFOs (`agents/claude_code/adapter.rs` `register_pending_ask_fifo` / `submit_ask_answer`). `mkfifo` has no Windows equivalent.

Crucial fact for scoping: **`#[cfg(unix)]` is true on macOS.** `UnixStream`, `UnixListener`, and `mkfifo` all exist and work on macOS, and the trait abstraction the seam introduces compiles unchanged. So the macOS port is *verify the unix path runs* + *introduce the seam* + *gate the few ungated unix-only spots so the crate can compile under `#[cfg(windows)]`* — **not a rewrite**. Windows named-pipe / loopback work is deferred.

**Peer-credential caveat — macOS is NOT `SO_PEERCRED`.** macOS has no `SO_PEERCRED`. `tokio`'s `UnixStream::peer_cred()` (already used at `mcp/transport.rs:117`) abstracts the platform difference: on Linux it reads `SO_PEERCRED`, on macOS it reads `LOCAL_PEERCRED` / `getpeereid` (the BSD `xucred`). The uid extraction therefore works on macOS through the SAME `peer_cred().uid()` call — but with two semantic differences the contract must respect: (a) macOS exposes **no peer PID** (`UCred::pid()` is `None`), so the boundary must rely on uid and never on a peer PID; (b) `LOCAL_PEERCRED` is a connect-time snapshot. The upshot: no macOS-specific peer-cred code is needed (tokio handles it), but the macOS verification MUST include an explicit **cross-uid rejection test**, not a same-user smoke walk, because a same-user walk exercises only the happy path and would never reveal a uid-extraction discrepancy.

**Socket base path is portable but NOT safe as-is — it must move into a per-user `0700` directory.** Both `config.rs:305` and `mcp/mod.rs:46` derive from `std::env::temp_dir()`, so no hardcoded `/tmp`. But on Linux `temp_dir()` is the shared, sticky-bit, world-writable `/tmp`, and the endpoints land there directly (`/tmp/nergal.sock`, `/tmp/nergal-mcp.sock`, `/tmp/nergal-plan-{pid}.fifo`) under predictable names. That exposes three concrete local attacks (see Decision 6): (1) a different-uid process squats the socket path → `remove_file` fails under the sticky bit → `bind` fails → persistent daemon DoS; (2) the create→`chmod 0600` window on the hook socket lets a different-uid process connect before perms tighten; (3) the plan-review / ask-user FIFOs have NO peer-auth at all — any local process can guess the PID-derived name and write a forged `allow` decision, forging the human approval gate on agent-spawned worktrees. macOS's `temp_dir()` is already a per-user `0700` `/var/folders/.../T`, so the threat model genuinely *differs by OS* — the earlier "no change needed, confirmed" claim was wrong. **Fix (Decision 6): place every endpoint inside a per-user `<temp_dir>/nergal-<uid>/` directory created `0700` and uid-validated before any bind.** This harmonises both platforms on the stronger (macOS-like) model.

## Goals / Non-Goals

**Goals:**

- Introduce a `PlatformListener` / `PlatformStream` seam that the hook server and MCP daemon use, so the Windows iteration drops in without touching the call sites or dispatch logic.
- Move (not rewrite) the existing Unix code behind the seam; zero behaviour change on Linux.
- Gate the remaining ungated Unix-only primitives (`hooks/server.rs` perms, `hooks/cli.rs` `unix::net` import + connects) behind `#[cfg(unix)]` so the crate compiles under a Windows target.
- Define the full cross-platform contract in the spec: per-platform peer auth, the unified blocking primitive, the Windows named-pipe/loopback target.
- Verify the real hook / MCP / cross-session flows on macOS.

**Non-Goals:**

- Implementing the Windows transport body (named-pipe or loopback). Deferred; only the seam + the contract land now.
- Migrating the plan-review / ask-user FIFOs off `mkfifo` for Linux/macOS. POSIX works on both; the unification is the documented target, implemented with the Windows path.
- Any frontend change (IPC is backend-only), any DB/migration, any new runtime dependency.
- The other multiplatform clusters (A3 `/proc`→sysinfo, B xdg/notify→Tauri plugins, C bundle+CI+signing). Tracked separately.

## Decisions

### Decision 1: A trait seam (`PlatformListener` / `PlatformStream`), not per-call-site `#[cfg]`

**Chosen:** Define two abstractions in a new `src-tauri/src/platform/ipc.rs` (or `ipc/mod.rs`) module — a listener that binds + accepts, and a stream that connects + reads/writes — with a Unix implementation that wraps today's `UnixListener`/`UnixStream`. The hook server and MCP daemon program against the abstraction.

**Alternatives considered:**

- *Sprinkle `#[cfg(unix)]` / `#[cfg(windows)]` at every call site.* Rejected: the connect sites alone are five points in `hooks/cli.rs` plus the MCP handlers; duplicating bind/accept/connect logic per-OS at each site is unmaintainable and is exactly what the MCP module doc already warns against. A single seam localises the per-OS code.
- *Adopt an existing crate (e.g. `interprocess`).* `interprocess` does offer a `LocalSocket` abstraction over unix-socket + named-pipe. Attractive, but (a) it adds a dependency and a migration of the already-working, already-`cfg`-gated MCP transport; (b) its framing/credential model differs from the hand-rolled length-framing + `SO_PEERCRED` we rely on; (c) the macOS iteration does not need it (unix path works). **Decision: hand-roll the thin seam now; re-evaluate `interprocess` if/when the Windows body proves it carries its weight.** Noted as an open question.

**Shape:** mirror the existing `UnixSocketTransport` (`mcp/transport.rs:70`), which already has the right surface (`bind` → `accept` → stream + peer identity, `connect`). Generalise its async-reader/writer-generic framing helpers — they are already platform-agnostic and need no change. The hook socket (newline-delimited, fire-and-forget, no response path) and the MCP socket (length-framed, request/response) are different protocols over the **same** transport primitive; the seam is the listener/stream, not the protocol.

### Decision 2: macOS-iteration cut — Unix impl + seam only; Windows deferred

The trait gets exactly one implementation this iteration (Unix). The Windows impl is a stub/absent module gated by `#[cfg(windows)]`. The crate must still **compile** under a Windows target (CI doesn't build Windows yet, but `cargo check --target` should not blow up on ungated `unix::net`), which is why the gating tasks (Decision 4) matter even before the Windows body exists.

Rationale: the user decision is explicit — macOS first, Windows deferred. Building the Windows body now would be speculative (named-pipe vs loopback is itself unresolved, Decision 3) and untestable without a Windows CI target.

### Decision 3: Windows transport — named-pipe vs TCP-loopback+token (deferred, recommendation recorded)

Both are viable; recording the trade-off so the deferred iteration starts from a decision, not a blank page.

| Aspect | Named pipe (`\\.\pipe\nergal-*`) | TCP loopback + token |
|---|---|---|
| Access control | OS-native security descriptor (owner-only ACL) — maps directly to today's uid wall | Any local process can connect to the port; a per-session secret token must be checked in-band |
| Peer authentication | `GetNamedPipeClientProcessId` / SID on the pipe — true peer identity, mirrors `SO_PEERCRED` | No OS peer identity; the token *is* the auth. Token must be delivered to the shim out-of-band (env var / file) and not leak |
| Tokio support | `tokio::net::windows::named_pipe` (in-tree, no new dep) | `tokio::net::TcpListener` (already used) |
| Path model | Pipe name string — analogous to a socket path | Host+ephemeral port; port must be discovered/published to the shim |
| Firewall / surface | No network surface | Binds a loopback port; AV/firewall noise possible |

**Recommendation:** named pipe. It preserves the security model we have on Unix (OS-enforced owner-only access + real peer identity via the pipe's client SID), needs no new dependency (`tokio` ships the named-pipe types), and maps cleanly onto the existing path-string abstraction (`std::env::temp_dir()`-style name → pipe name). Loopback+token is the fallback only if a named-pipe edge case (e.g. a sandboxed agent CLI that can't open pipes) forces it; it trades OS-enforced ACL for an in-band secret we'd have to manage and protect. The MCP socket is security-sensitive (today: `0600` + uid check), so the ACL-backed option is the safer default.

**Loopback fallback is constrained, not blessed unconditionally.** A bare "present a token on connect" contract is materially weaker than the uid wall — an implementer could mint a weak/predictable token, ship it via a world-readable file, and still satisfy a loose scenario. The spec therefore makes the loopback fallback conditional on ALL of: (a) ≥128-bit CSPRNG token; (b) delivery over a channel not readable by other local users (NOT a world-readable file, NOT an inspectable process env var); (c) constant-time comparison; (d) reject-before-dispatch (validate the token before reading any request payload). If those cannot be met, named-pipe is required. The binding artifact for this is the spec's "Constrained Windows loopback fallback" requirement — recording it here so the deferred Windows iteration cannot silently land a forgeable boundary.

### Decision 4: Gate the ungated Unix-only spots

Two spots are currently ungated and will break a Windows compile:

- `hooks/server.rs:206-209` — `set_permissions(0o600)` in a bare `{ }` block. Wrap in `#[cfg(unix)]` (the MCP transport at `:83` is the exact pattern to copy).
- `hooks/cli.rs:3` — top-level `use std::os::unix::net::UnixStream;` and the five connect sites. These move behind the seam (`PlatformStream::connect`); the `unix::net` import disappears from the call site and lives only inside the Unix impl module.

The MCP transport's perms (`:83`) and `peer_uid` (`:115-124`, with a `#[cfg(not(unix))]` stub) are already gated — leave them, they are the template.

### Decision 5: Unify the blocking FIFOs onto the primitive — contract now, Linux/macOS keep FIFO

The blocking plan-review / ask-user round-trips are the one place that uses `mkfifo`, which has no Windows analogue. The spec defines that these round-trips move onto the `PlatformStream` primitive (a short-lived per-request connection carrying the decision/answer back), unifying the mechanism. **But for the macOS iteration the existing FIFO path stays** — POSIX `mkfifo` works on both Linux and macOS, the code is proven, and a full rewrite buys nothing for the macOS target. The unification is sequenced with the Windows iteration, where it becomes necessary. The seam is designed so this is a drop-in: the blocking response is just another `PlatformStream` round-trip.

**Security hardening that does NOT wait for the Windows unification.** The FIFO carries an authorization decision — plan-review feeds the human approval gate on agent-spawned worktrees — so leaving it as the proven-but-unauthenticated path is unacceptable even for the macOS cut. Three hardenings apply to the FIFO *now*, because they are cheap and close a real local-attacker hole (an attacker who guesses the PID-derived name and writes a forged `allow`):

1. **Owner-only boundary.** The FIFO moves into the per-user `0700` IPC directory (Decision 6) so only a same-uid process can open it for writing. This is the structural fix for the forged-approval attack and is the same one-line relocation the sockets get.
2. **Liveness-aware deny — implemented with FIFO-compatible primitives, NOT a connection death signal.** The blocking `read_to_string` (`cli.rs:188`) is unbounded today — a crashed GUI hangs the agent loop forever. But plan review is a human-in-the-loop gate that *legitimately* sits pending for minutes; a blunt short timer that force-denies mid-deliberation silently rejects the plan while the human is still deciding (a correctness/UX regression). The wait must distinguish **GUI death** (resolve to a safe deny) from a **live GUI with a pending human decision** (keep waiting). **Crucial constraint:** the macOS-iteration cut keeps the POSIX **FIFO**, which is *connectionless* — during deliberation no writer is attached, so there is NO "writer end closed"/EOF signal to observe. A death signal that only exists on a connected `PlatformStream` (the deferred unified target) cannot be used here. The FIFO-path mechanism is therefore explicit:

- **The GUI writes an identity token, not just a pid — and only the `flock`-holding instance writes it.** On startup the GUI writes `gui.pid` into the per-user IPC directory containing its pid AND a process-start-time token (Linux: the `starttime` field of `/proc/<pid>/stat`; macOS: the start time from `proc_pidinfo`/`kinfo_proc`). A bare pid is insufficient: if the GUI crashes and the kernel recycles its pid to an unrelated process during deliberation, `kill(gui_pid, 0)` returns `0` and the CLI misreads "GUI alive" → the death goes undetected and the wait degrades to the wall-clock backstop, silently defeating the fast-detection the mechanism exists for. The start-time token disambiguates a reused pid (different start time ⇒ not our GUI ⇒ dead). On Linux a `pidfd` is the cleaner alternative and SHOULD be preferred where available. **Ownership constraint (reconcile with the Decision-6 `flock`):** `gui.pid` SHALL be written and refreshed ONLY by the instance that holds the IPC `flock` / owns the IPC surface — never by a second same-uid instance that started, lost the `flock`, and deferred. Otherwise a launch-race sibling could overwrite `gui.pid` with its own (about-to-exit) pid, and a later CLI would read a dead pid → spurious (safe) deny while the real server is alive. The token granularity residual — a pid recycled within the same `starttime` clock tick (~10 ms) would present an identical pid+token — is astronomically improbable and degrades only to the human-scale backstop, never to a forged approval; the backstop is the final bound.
- **The CLI poll is timer-driven, not edge-driven.** The hook CLI opens the FIFO **non-blocking** (`O_RDONLY | O_NONBLOCK`) and `poll()`s it with an explicit bounded timeout (≈1 s) that DRIVES the liveness cadence — NOT `poll(-1)`. A read-only FIFO with no connected writer (the normal deliberation state) returns `POLLHUP` continuously on Linux and with BSD/macOS-divergent semantics; an infinite-wait or POLLHUP-edge loop busy-spins a core / hammers `kill()` / or never wakes to run the liveness check. The CLI ignores `POLLHUP` (no-writer is expected), `read`s on `POLLIN`, and on each timeout tick checks GUI liveness via the pid+start-time token: token-mismatch or `ESRCH` ⇒ GUI gone ⇒ safe deny.
- A human-scale wall-clock bound (minutes, configurable) is the final backstop and SHOULD surface a re-arm rather than a silent deny.

Net: deny on detected GUI death (pid+start-time token, timer-driven poll), keep waiting while the GUI is alive, never silently deny mid-deliberation. When the FIFO is later migrated onto `PlatformStream` (Windows iteration), the connection EOF/`ECONNRESET` replaces the pidfile poll. Do NOT specify a connection death signal for the FIFO cut.
3. **Pre-seed / stale guard + RAII cleanup.** The FIFO is unlinked on entry and on exit; a pre-existing FIFO at the expected path is treated as hostile (removed, not reused) so a stale decision from a crashed run or a recycled PID cannot be read as a live answer.

Rationale for not doing the *full `PlatformStream` unification* now: scope discipline. The macOS acceptance criterion is met without rewriting the FIFO mechanism. But the three hardenings above are not the unification — they are the minimum to make the approval gate non-forgeable on the path we ARE shipping. The full unification (FIFO → `PlatformStream`) is still sequenced with the Windows iteration.

### Decision 6: All IPC endpoints live in a per-user runtime directory rooted at an UN-SQUATTABLE base (NOT `/tmp`)

**Problem:** on Linux, `temp_dir()` is the shared, sticky-bit, world-writable `/tmp`, and the endpoints land there directly under predictable names. This is the root cause of three otherwise-separate findings: the stale-socket squat DoS (a different-uid process pre-creates `/tmp/nergal-mcp.sock`; our `remove_file` fails under the sticky bit; `bind` fails; daemon won't start, trivially re-armed), the hook-socket create→`chmod 0600` TOCTOU window (a different-uid process can `connect()` in the interval before perms tighten), and the forgeable plan-review FIFO (no peer-auth; any local process guesses the PID-derived name and writes a forged `allow`). macOS's `temp_dir()` is already per-user `0700`, so these are Linux-specific — meaning the threat model differs by OS, which the original "no change needed, confirmed" claim missed.

**Critical subtlety (do not regress this):** a per-user directory whose *parent* is shared sticky `/tmp` does NOT fix the squat — it relocates it. The sticky bit stops a different uid from *deleting* the victim's `/tmp/nergal-<uid>/`, but NOT from `mkdir`-ing it *first*, owned by the attacker. The victim then finds it foreign-owned, refuses to bind (correct), and cannot remove it (sticky bit) — a persistent, trivially re-armed, cross-uid denial of the **entire** IPC surface. The uid is public (`id`, `/etc/passwd`), so the leaf name is guessable. The fix MUST therefore root the per-user directory at a base into which **no other uid can write at all**, not merely a `0700` leaf under a shared parent.

**Chosen — per-platform un-squattable root:**

- **Linux:** root at `/run/user/<uid>/nergal/`, where `<uid>` is computed from **`getuid()`** — NOT read from the `$XDG_RUNTIME_DIR` environment variable, and with **no `$XDG_RUNTIME_DIR` override at all** (dropped — see below). This is load-bearing: the three processes that must derive the SAME path (GUI bind, MCP shim connect, hook-CLI connect) do not share the same env — project memory records that **Codex sanitizes the MCP server env** (the shim already recovers `NERGAL_SESSION_ID` from a `/proc/<pid>/environ` ancestor). If the root were keyed off `$XDG_RUNTIME_DIR`, a value present for the GUI but stripped for the shim would send the shim's resolver down the fallback while the GUI bound under `/run/user/<uid>/` → connect targets a dead path → MCP silently broken on a supported agent. `getuid()` is identical in every process regardless of env stripping, so the canonical path is stable. We deliberately do NOT consult `$XDG_RUNTIME_DIR` even as an override: an override whose validation is weaker than string-equality would reopen the exact divergence for any user with a non-standard `$XDG_RUNTIME_DIR` (GUI validates+binds the custom path; env-stripped shim computes `/run/user/<uid>` → connect to a dead path), and a string-equal override is a provable no-op. It adds no capability `getuid()` lacks, so it is omitted.
- **Linux fallback** (when `/run/user/<uid>` does not exist / fails validation — minimal/non-systemd hosts, some SSH sessions): a `0700` directory under the user's home derived from **`getpwuid(getuid())->pw_dir`**, NOT the `$HOME` env var (under `sudo`/`su` `$HOME` frequently still points at the original user's home → cross-uid write or divergence). The fallback's parent SHALL be validated owned by `getuid()` before use, with a clear refusal rather than a confusing permission error. Never a guessable-named directory under shared `/tmp`.
- **macOS:** `temp_dir()` is already a per-user `0700` `/var/folders/<hash>/T` whose parent is user-owned — genuinely un-squattable — so `temp_dir()/nergal/` is safe here.

Place EVERY endpoint (hook socket, MCP socket, plan/ask FIFOs) inside that directory. Validate on startup: the directory SHALL be owned by the current uid and no broader than `0700`; on a foreign-owned or over-permissive directory the system SHALL refuse to start the affected IPC surface and log. With an un-squattable root the refuse-and-log branch is unreachable by another uid in normal operation; it remains a tripwire, not the mitigation. This closes all three findings at the root and harmonises both platforms on the stronger model. The per-socket `0600` chmod stays as defence in depth.

**Concurrent same-uid instances:** the un-squattable root excludes *different*-uid interference, but two same-uid launches (two windows, relaunch-before-exit) still race — instance B's "remove stale endpoint then bind" would unlink the socket instance A is actively serving (split-brain). Guard with an `flock`'d lockfile in the per-user dir plus probe-then-adopt-or-fail. The liveness predicate is **`connect()` success vs `ECONNREFUSED`**, NOT a reply: the hook socket is fire-and-forget (`server.rs:220`, no response path) so a live peer never "answers" — a connectable socket means an instance is live (defer to it); `ECONNREFUSED` on an existing socket file means a stale endpoint with no listener (provably dead → safe to unlink). Stale-endpoint removal is conditional on that probe, never unconditional.

**`/run/user/<uid>` teardown edge:** systemd removes `/run/user/<uid>` on last-session logout, so a long-running instance across a logout→login churn can find its IPC dir gone mid-run. The server SHALL treat a vanished IPC dir as a re-arm condition by **re-running the full resolver** (NOT "mkdir `/run/user/<uid>`" — after teardown `/run/user` is root-owned `0755` and a non-root process gets `EACCES`; re-running the resolver correctly falls to the `getpwuid` home fallback) rather than dying silently. Accepted churn-window caveat: while the GUI sits on the home fallback during the logout gap, a re-login recreates `/run/user/<uid>` and any post-login agent-spawned shim/CLI computes `/run/user/<uid>` → a transient bind/connect divergence across the churn until the instance restarts. This is an availability edge the `/tmp` location did not have, accepted (and documented) as the cost of the un-squattable root.

**Single shared path resolver:** the directory resolution AND the `sun_path` overflow/short-hash logic (see the macOS risk) MUST live in one shared resolver used by all three processes that touch the paths — the GUI (bind), the MCP shim (connect), and the separate `hooks/cli.rs` binary (connect + FIFO). Reimplementing the branch per-call-site would let the connect side silently target a different path than the bind side. One resolver, no duplication.

**Cost:** the endpoint paths change (`/tmp/nergal.sock` to `$XDG_RUNTIME_DIR/nergal/hook.sock`). Endpoints are ephemeral runtime artifacts recreated each launch (not persisted state) and the single shared resolver guarantees both sides agree, so the relocation is transparent and needs no migration. This is the single intentional observable change to the "No Linux regression" contract, called out explicitly there.

### Decision 7: Verify the boundary with an automated cross-uid test, and log every boundary event

**Problem:** the uid wall is "the only enforced MCP access control" yet nothing currently fails if the seam refactor silently drops it — clippy, the unit suite, fmt, tsc, and a same-user manual walk all stay green. And a bypassed or broken boundary is today indistinguishable from a quiet system (no audit logging).

**Chosen — TWO distinct tests, do not conflate them:**

- **(a1) Comparison-branch test (CI, mocked).** A test that injects a foreign uid into the peer-credential comparison (mock `peer_uid`) and asserts rejection. This catches the regression we care about most — a refactor silently dropping the `!=` check — and runs in CI with no privileged setup. **It does NOT, and is NOT claimed to, verify the OS extraction** (`peer_cred().uid()` returning the real foreign uid), because it replaces that very call.
- **(a2) Real foreign-uid extraction test (macOS + Linux acceptance, privileged).** A genuine cross-uid connection that asserts `peer_cred().uid()` returns the real foreign uid via `LOCAL_PEERCRED`/`getpeereid` (macOS) / `SO_PEERCRED` (Linux). **Crucial test-design constraint (do not get a false green):** the *production* socket lives inside the Decision-6 `0700` per-user dir, which a foreign uid (`nobody`) **cannot traverse** — `connect()` returns `EACCES` at the directory *before* `accept()`/`peer_cred()` ever runs. A harness pointed at the production socket would see `connect` fail and wrongly conclude "boundary rejected ✓", proving only the directory perms (already known) while leaving the `peer_cred` path **unexercised but believed-verified** — the round-2 failure mode reintroduced. So a2 MUST bind a **dedicated test socket in a deliberately-traversable parent** (e.g. a `0711`/`0755` tmp dir) so the foreign peer actually reaches `accept()` and `peer_cred()` runs; the assertion is then `peer_cred().uid()` == nobody's uid AND the boundary rejects post-accept. The harness MUST distinguish `EACCES`-at-`connect` (the dir blocked the peer — `peer_cred` NOT exercised → test INVALID, not a pass) from `accept()`+peer-cred-rejection (the thing under test). State plainly: on the shipping `0700`-dir config the directory IS the real boundary and `peer_cred` is defence-in-depth, verified here against a relaxed-perm test socket — an `EACCES` must never masquerade as a peer-cred rejection.
- **Committed harness:** run a tiny connector binary as the always-present unprivileged `nobody` account via `sudo -u nobody` — `nobody` exists on stock macOS and Linux, needs no account provisioning, and a setuid helper (a shipped security liability) is explicitly NOT chosen. **Stage the connector binary in a world-execable location** (`/tmp`, `/usr/local/bin`) before `sudo -u nobody`: on stock macOS `/Users/<dev>` is `0700`/ACL-restricted, so `nobody` cannot traverse it to `exec` a connector built under the dev's home/`target/` (`EACCES` at exec — distinct from a peer-cred rejection and a connect-`EACCES`, and easy to mis-bucket). An exec failure means the harness is not runnable → UNVERIFIED-pending, NOT a boundary pass. If `sudo` is unavailable on the target box, the macOS extraction result SHALL likewise be recorded as **explicitly UNVERIFIED-pending** (a tracked gap), NOT silently treated as covered by the (a1) mock — because (a2) is the only thing that exercises `LOCAL_PEERCRED`, an un-run (a2) means the macOS boundary is unverified and must be surfaced as such, not checked off.

Conflating these — claiming the mocked (a1) test "confirms `peer_cred()` returns the real foreign uid" — would leave the macOS boundary unverified while believing it verified. Keep them separate.

**(b) Audit logging, rate-limited.** Emit a logged audit line for every rejected peer (with uid), every bind failure, every IPC-directory validation refusal, and every blocking-wait timeout; make the best-effort notifier sends log on failure instead of swallowing the error. **But rate-limit/coalesce** repeated rejections from the same uid (e.g. count + first/last timestamp per window) — an unbounded "log every rejected connect" is itself a local DoS: a hostile uid spams rejected connects to fill the disk / thrash rotation / drown the real signal. Observability must not become an attack amplifier.

## Risks / Trade-offs

- **[Seam leaks Unix assumptions]** → Model the trait on `UnixSocketTransport`, whose surface (`bind`/`accept`→(stream, peer-id)/`connect`) is already platform-neutral and whose author already intended a Windows drop-in. Peer identity is returned as an opaque value (uid today; pipe SID later), not a `u32` baked into the signature where avoidable.
- **[Hook socket vs MCP socket are different protocols]** → The seam abstracts the *transport* (listener/stream), not the protocol. Newline-delimited fire-and-forget (hook) and length-framed request/response (MCP) both ride the same `PlatformStream`; no attempt to unify the protocols, only the primitive.
- **[Windows compile rot]** → Without a Windows CI target, ungated `unix::net` can creep back. Mitigation: the gating tasks land the `#[cfg(unix)]` discipline now; a follow-up can add `cargo check --target x86_64-pc-windows-msvc` to CI when the Windows body starts.
- **[Hand-rolled seam later duplicates `interprocess`]** → Accepted: the seam is thin (≈ one module). If the Windows body shows the abstraction is non-trivial, swapping the internals for `interprocess` is localised to the impl module, not the call sites. Recorded as an open question.
- **[macOS behavioural surprise despite `#[cfg(unix)]`]** → `mkfifo`, peer-cred, `temp_dir()` are POSIX/portable but macOS edge cases exist: (a) peer-cred is `LOCAL_PEERCRED`/`getpeereid`, not `SO_PEERCRED`, with no peer PID — tokio abstracts the uid but the boundary must not depend on a PID; (b) `temp_dir()` returns a long per-user `/var/folders/...` path — `sun_path` ~104 bytes on macOS vs ~108 on Linux, AND the per-user `0700` dir from Decision 6 adds a path component, tightening the budget further. Mitigation: the seam enforces the `sun_path` bound at bind time (deterministic short hash of the endpoint name) rather than relying on a human noticing; explicit macOS cross-uid verification task (Decision 7), not a same-user smoke walk.
- **[Per-user dir adds path length on macOS]** → Decision 6's `<temp_dir>/nergal-<uid>/` prefix eats into the already-tight macOS `sun_path` budget. Mitigation: keep the per-user dir name short (`nergal-<uid>`) and the endpoint filenames short (`hook.sock`, `mcp.sock`); the bind-time length check (above) is the backstop.

## Migration Plan

1. Land the seam module + Unix impl (wrapping existing code), no call-site behaviour change.
2. Add the per-user `0700` IPC directory (Decision 6): create + uid-validate on startup, repoint every endpoint path (sockets + FIFOs) into it.
3. Repoint hook server + MCP daemon + hook CLI connect sites through the seam.
4. Gate the ungated Unix-only spots (`hooks/server.rs` perms, `hooks/cli.rs` import); add the hook-socket peer-uid check; add the FIFO timeout-to-deny + RAII/pre-seed guard; add the audit logging (Decision 7).
5. Add the automated cross-uid rejection test (Decision 7).
6. Verify Linux full-check green (no regression except the intentional endpoint relocation) — this is the rollback gate.
7. Verify the real flows on macOS, including the cross-uid rejection test and the `sun_path` length check under the per-user dir.
8. (Deferred, separate iteration) Implement the Windows named-pipe impl + migrate the blocking FIFOs onto the primitive.

Rollback: the change is additive + a move + endpoint relocation; reverting restores the direct `unix::net` usage and the old `/tmp` paths. Endpoints are ephemeral (recreated each launch), so no data/schema is involved and rollback is code-only. The one caveat: a running pre-rollback daemon binds the new path while a post-rollback build binds the old path — but since both are per-launch and the installer restarts the app, there is no split-brain across the upgrade boundary.

## Open Questions

- **`interprocess` crate vs hand-rolled seam for the Windows body** — re-evaluate when the Windows impl starts; the seam is designed so the decision is localised to the impl module.
- **Peer-identity type in the trait signature** — uid (`u32`) on Unix vs a pipe client SID on Windows. Resolve when the Windows body lands; for now the Unix impl returns the uid as it does today, and the trait can expose an opaque `PeerId` if needed. (macOS exposes no peer PID — the opaque type must not require one.)
- **macOS `sun_path` length** — RESOLVED as a contract: the seam enforces the bound at bind time (deterministic short hash when the per-user-dir-prefixed path would overflow), rather than leaving it to a verification-time observation. The macOS task still confirms the real resolved path binds.
- **Whether the hook socket should also adopt length-framing** when unified — out of scope; the hook protocol stays newline-delimited fire-and-forget, only the transport primitive is shared.
