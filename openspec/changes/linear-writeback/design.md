# Design — linear-writeback

## Context

The mirror (`linear-mirror`) is a durable local copy kept fresh by a bounded poller; the binding (`linear-agent-integration`) ties a session to its issue via `active_linear_issue_id`. This change adds the write direction, mirroring the archived `clickup-writeback` adapted to Linear's GraphQL API. The hard parts are not the API calls — they are (a) keeping optimistic UI, the durable mirror, and polling from corrupting each other, (b) the fact that comments are a fundamentally different write than fields, and (c) making outward-facing writes against a full-access personal key structurally safe, not UI-guarded.

The `clickup-writeback` design (archived, 7 decisions + 1 revision) is the reference. The decisions below carry over with Linear-specific adaptations; where Linear diverges materially from ClickUp it is called out explicitly so the iprev does not flag a copied-but-wrong assumption.

## Decision 1 — Optimistic UI is a frontend overlay; the durable mirror holds only server-acked truth

**Decision**: an edit updates an **in-memory frontend overlay** (a pending-edit atom in `stores/linear.ts`, mirroring `clickupOverlayAtom`) for instant UI, but **does NOT mutate the durable SQLite mirror until the API acknowledges**. The mirror always holds server truth. On API success, the mirror is updated by the normal reconcile and the overlay clears; on failure, the overlay reverts + toast.

**Why**: if the optimistic value were written to the durable mirror before the API ack, a daemon crash in that window would leave the mirror holding a value never sent upstream, and the next poll would silently overwrite it with the stale server value — violating "never silently clobber" with no registry left to detect it. Keeping optimism in the volatile UI layer means a crash loses at most an un-acked edit (the user re-does it) and the durable mirror is never wrong.

**Alternatives considered**:
- *Optimistic write to the durable mirror + in-memory rollback registry*: the split-brain above. Rejected.
- *Durable `pending_writes` table + dirty flag on mirror rows*: correct but adds a migration and crash-reconcile logic for a single-user tool; the frontend-overlay model achieves the same safety more simply. Rejected as over-built.
- *Write-through (no optimism)*: laggy UX. Rejected.

## Decision 2 — Echo / conflict by value comparison, evaluated in the run loop (not inside `reconcile`)

**Decision**: `updatedAt` is a timestamp on the **issue**, not per-field, not predictable pre-call — so it is used only as a coarse "this issue changed, re-examine it" trigger, never as the echo key or per-field order. Echo/conflict are decided by **value comparison against the fetched issue payload** (the `cycle` the run loop already holds, carrying each issue's current `state.id` and `assignee.id`):

- Before a write's API call, record a **provisional** entry `(issue, field, written_value, pre_write_value, at)` in an in-memory `recent_writes` map (`linear/writeback.rs`, a `WritebackRegistry` analog). On API **failure**, clear the entry (and the frontend overlay reverts). On success it stays (refreshing `at`). Recording *before* the call closes the TOCTOU window (review #7): a poll that lands between the write hitting Linear and the command resuming would otherwise see the new server value with no registry entry and mis-fire.
- On the next poll, for each recent-write entry, compare the **server's current value** of that field — read from the **post-reconcile mirror** (see below) — to: (a) our written `value` → match ⇒ **own echo**, clear the entry, suppress; (b) equals the pre-write value ⇒ our write hasn't landed yet ⇒ unrelated; (c) neither ⇒ someone else changed it ⇒ **conflict** (Decision 3).

**Where this runs + where the data comes from (reviews #6, N2, N3 — critical correction)**: `reconcile(conn, cycle, gen)` (`poller.rs:159`) is a **pure, generation-guarded, heavily-unit-tested** function, and the run-loop wrapper `run_cycle(client, db)` (`poller.rs:344`) **constructs `FetchedCycle` locally and drops it on return — it returns only a `ReconcileOutcome`** (`poller.rs:47`, fields `discarded/tombstoned/evicted/newly_assigned/baseline_set/fetch_error`, NO per-issue field values). So at `mod.rs:1033` neither the fetched `cycle` nor the per-issue server values are in scope — the original "compare against the fetched cycle in the run loop" was unimplementable. **Resolution (option b): re-read the just-committed mirror.** After `run_cycle` returns, the run loop (which holds the `AppHandle` + `&db`) reads, for each registry entry, the issue's **current mirror value** (`state_id`/`assignee_id`) — which IS server truth, because `reconcile` blind-upserts the fetched payload into the mirror. This needs no change to `reconcile`'s purity or `run_cycle`'s return type. The `WritebackRegistry` is reached from the spawned poller task via `app.state::<WritebackRegistry>()` (NOT `tauri::State` injection — that is command-only; the poller loop is a spawned task, review N3), so it must be `manage()`d AND fetched through the app handle there. The two outputs: (1) the assignee own-echo filter removes ids from `outcome.newly_assigned` before `notify_assignments`; (2) a divergent state/assignee value emits `linear:write-conflict`. The real contract is "**before the notification**," not "before the set-diff," and the comparison source is the **post-reconcile mirror**, not the dropped cycle.

**Edge (mirror-read vs cycle):** if an issue we wrote to fell out of the poll window (tombstoned/evicted, not upserted), its mirror row is stale/absent — but then there is no fresh remote value to conflict against either, so skipping it is correct. An issue we just assigned to ourselves is in-window by construction (viewer-assigned issues are always fetched, `client.rs viewer_assigned_issues`).

**Linear divergence (verified)**: the Linear poller does **blind upserts** and its *only* notification today is `newly_assigned → linear:assigned` (`mod.rs:1033`/`:1111`); a state change emits no per-field event, just the generic `linear:changed`. Two consequences:

1. **Assignee echo** is the genuine suppression: filter an own "assign to me" out of `outcome.newly_assigned` in the run loop **before** `notify_assignments`, or the user's own assignment self-fires `linear:assigned`. Pinned with a regression test.
2. **State has no notification to suppress** (review #4, #12) — so "state echo" is not about silencing a notification; it is about deciding whether a state divergence is the user's own echo (suppress the conflict warning, clear the entry) or a genuine remote conflict (Decision 3 → emit a NEW `linear:write-conflict` event). The conflict event + the run-loop comparison are the real new machinery this change adds; the Linear poller surfaces no per-issue conflict today.

## Decision 3 — Conflict resolution is "server-wins + warn-on-divergence", scalar-only (no additive merge path)

**Decision** (review #5 — corrected from a copied ClickUp misnomer): the in-scope write fields — workflow state and assignee — are **both scalar single-value** fields. Because optimism lives ONLY in the frontend overlay (Decision 1), the durable mirror never holds the local edit and the poller **blind-upserts server truth** (`poller.rs:200`). So there is no code path where the local value "wins" and **no `updatedAt` comparison ever happens** — calling it "last-writer-wins by `updatedAt`" would push an implementer to add a pointless timestamp gate. The actual resolution is: **the server value always lands in the mirror (blind upsert); the system warns IFF that server value diverges from what the user just wrote** (i.e. a real remote writer superseded the local edit). The own-echo case (server == written) is suppressed silently; the unrelated case (server == pre-write, our write not yet landed) does nothing. Never silently clobber a divergence — emit `linear:write-conflict` (`{issue_id, field, your_value, remote_value}`) and toast.

**Linear divergence**: ClickUp's Decision 3 split fields into scalar (LWW+warn) vs additive (assignees/checklist/labels → merge-no-warn). **Linear has no additive field in this change's write surface**: assignee is single (not a set — `assignee_id: Option<String>`, verified `mirror.rs:537`), and labels/description/priority are out of scope. So the additive-merge branch is intentionally absent. If a future change adds label writeback (a true set), the additive branch returns — noted so the absence reads as a scoping decision, not an oversight.

## Decision 4 — Comments are a separate post-once model, not part of the optimistic/rollback pipeline

**Decision**: a comment is not an issue field — it lives on a separate resource, posting it does **not** change `state.id`/`assignee.id`, it is **append-only (cannot be rolled back)**, and a timeout-but-succeeded post retried (or the closure firing twice) posts a **duplicate team-visible comment**. So comments get their own path:

- **No optimistic mirror insert**: the comment is inserted into `linear_comments` only after `commentCreate` returns the created comment id.
- **Post-once with ambiguity handling**: on a clear success, store it. On an ambiguous failure (timeout, no response), **do not auto-retry**; mark the send "uncertain" and, before any retry, re-fetch the issue's comments and check whether it landed (best-effort idempotency, since Linear has no comment idempotency key) — surface the uncertain state rather than risk a duplicate.
- **Echo**: a posted comment appears in the next live detail fetch; match it to the local pending comment by author+body+timestamp to avoid re-rendering it as "new."

**Linear divergence (verified)**: Linear comments are **not mirrored by the poller** — the `linear_comments` table exists but only `assemble_linear_context` reads it; the detail fetches comments **live** via `issue_detail` (`client.rs:324`, `comments(first:100)`). So the ambiguous-failure re-check re-fetches via the live `issue_detail` path (not a mirror read), and the post-success insert writes into `linear_comments` directly (so the context composer and the next detail open both see it). The comment author id is a **String** UUID (the viewer id), not ClickUp's `i64`.

**All comment writes route through the token gate — there is NO standalone comment command (review #1)**: this is a sharp divergence from how the detail's *direct* state/assignee controls work. A comment is auto-postable, team-visible, and irreversible, so — exactly as `clickup/writeback.rs:19-24` states ("`post_comment`/`verify_comment_landed` are NOT Tauri commands… wrapped behind the confirmation-token gate") — the detail's "post a comment" composer does NOT get its own un-gated `linear_post_comment` command. An implementer must NOT add a direct comment command mirroring the adjacent direct state/assignee commands; doing so breaches the exact boundary this change exists to defend.

**The comment-only token path does NOT close out the issue (review N1)**: a plain comment post and a closure are two distinct token kinds — collapsing them would make "post a quick comment" silently unbind the session and stamp it worked-and-closed (a destructive side effect). So the token carries a `close_out: bool` discriminant and there are **two request commands**: `request_comment_token(issue, comment)` (issues `close_out=false`, `state=None`) for the detail composer, and `request_closure_token(issue, state?, comment?)` (issues `close_out=true`) for the closure prompt. A single token-gated `execute_gated_write(token)` performs state (if present) + comment (if present), and performs the unbind + worked-closed marker **only when `close_out=true`**. Both paths are token-gated (security boundary intact); only the closure path mutates the binding. (Direct state/assignee edits stay un-token-gated per Decision 5's reversible-write carve-out; comments do not, because they cannot be un-posted.)

## Decision 5 — Structural write boundary: confirm → token → execute, plus server-side validation

**Decision**: the safety of an outward-facing write against a full-access personal key SHALL NOT rest on a React-side confirmation. Per-write Tauri commands registered in the invoke handler are directly callable from the webview, so a renderer bug or injected content could invoke them.

- **The closure and comment writes** (auto-fireable / irreversible / team-visible — the sharp edges) are **structurally gated**: the sole execution entry is a command that requires a short-lived **confirmation token** issued by the backend only when the user confirms the prompt for a specific `(issue, state?, comment?)`. A renderer cannot fabricate a valid token without going through the confirm step. This is a near-1:1 port of `clickup/closure.rs`'s `ClosureTokenStore` (`Mutex<HashMap<token, ClosureToken>>`, 30s TTL, destructive single-use `take`).
- **Routine state/assignee writes validate inputs server-side**: a state write checks the value is a real workflow state of the issue's team (`linear_workflow_states WHERE team_id = ?`), at the command boundary (not just hidden in UI). Linear has no computed fields to reject.

**Alternatives considered**:
- *UI-only confirmation*: rejected — exactly the gap iprev forced closed on `agent-spawned-worktrees` and `clickup-writeback`.
- *Token handshake on every field edit*: rejected as disproportionate — routine user-initiated state/assignee edits are reversible and low-blast-radius; the token gate is reserved for the auto-fireable closure and irreversible comments, with server-side validation covering the rest. (Accepted risk, carried from clickup: a renderer bug could fire a *reversible* state/assignee change against the full-access key.)

## Decision 6 — Write-back closure: ship-success + manual verb, explicit, idempotent

**Decision**: the closure is **not** anchored to a session-status transition. The `clickup-writeback` Revision 1 already falsified that anchor (nothing writes runtime session status transitions; `db.update_session_status` has zero callers), and Linear inherits the same runtime — so this change starts from the correct anchors directly, no revision needed:

1. **Manual verb (the primary path)**: "Close out issue" in the floating detail toolbar opens the closure prompt without a PR prefill. The user's actual workflow creates PRs in-prompt (agent-driven), so the manual verb is the main entry. Bound to `KeyC` (free in Linear's `VERB_KEYS` = S/W/P/B/R, verified `LinearTaskDetail.tsx:57`).
2. **Ship-success offer (automatic)**: when `git_ship` succeeds (PR created) for a session with an `active_linear_issue_id`, the frontend ship flow (`ShipDialog.tsx` success path) raises the closure offer — issue identifier, state picker from its team workflow states, comment composer prefilled with the PR link (user-editable). One offer per successful ship invocation (discrete event — no edge/level bookkeeping); a later re-ship offers again, intentionally. Push-only (no PR) does NOT offer.

**Write order + local close-out timing (review #8, N1)**: `execute_gated_write` performs, in order, (1) the **state** write (the reversible/retryable half — first, mirroring `clickup/closure.rs:24-32`), (2) the **comment** post (irreversible — second), then (3) **only when the token's `close_out=true`**, the local close-out (unbind + durable marker). For a `close_out=false` comment-only token (the detail composer), steps 1/3 are skipped — it posts the comment and nothing else, leaving the binding intact. When `close_out=true`, the local close-out is the core act and is applied **regardless** of the Linear outcomes (a failed state write still closes the issue out locally). Because the unbind removes the binding, the partial-failure retry surface ("state change failed — retry state?") MUST carry the `issue_id` explicitly — the session is already unbound (and for a ship-triggered closure the session keeps running), so the retry cannot re-derive the issue from the binding. The durable "worked & closed" marker is migration `026_linear_closed_out.sql` (`issue_id TEXT PRIMARY KEY, closed_at INTEGER` — identical to `019_clickup_closed_out.sql`), surfaced as a panel badge read on every mirror refresh (`linearClosedOutAtom`). Close-out happens even with neither Linear write half selected.

**Both Linear write halves are OPTIONAL and independent**: a state move (picker over the issue's team states — per-team custom, never hardcoded) and a comment composer; confirm executes only what was selected. State-only is the expected common case; an empty comment posts nothing. Selecting neither still closes out locally. No comment is ever sent "by default."

**Agent-agnostic constraint (recorded)**: this change installs no hooks, writes nothing to any agent's user settings, and hardcodes no agent. `git_ship` is Nergal's own command and works identically for CC/Codex/OpenCode/Pi sessions.

**Alternatives considered**:
- *SessionEnd hook as trigger*: rejected — noisy (restarts via `--continue`, accidental tab closes, the known close+reopen bug would all fire false offers) and rides the hook pipeline, the coupling this change avoids.
- *Automatic offer on worktree merge/remove*: deferred — merge often happens long after the work session; ship is the natural moment; the manual verb covers late closure.

## Decision 8 — Mutation responses must be checked for `success`; reads must exclude synthetic states; assign-to-me must have a real viewer id

Three command-boundary correctness guards the generic `execute::<T>` helper does NOT provide for free:

- **`success: Boolean!` (review #2)**: Linear's `issueUpdate`/`commentCreate` payloads carry a `success` boolean, and `execute` only bails on a GraphQL `errors` array (`client.rs:122-130`) — a `{"data":{"issueUpdate":{"success":false}}}` with no `errors` is classified `Ok`. So the mutations SHALL select `success` (and `issueUpdate` the updated `issue { state { id } assignee { id } }`) and the command SHALL treat `success != true` as a write failure (overlay reverts, registry entry cleared) — never a silent no-op the user believes landed.
- **Synthetic state exclusion (review #3)**: `linear_workflow_states` carries a `synthetic` flag; `ensure_state_placeholder` (`mirror.rs:67`) inserts `synthetic=1` placeholder rows with name `"(state {id})"` on paging skew. `read_team_states` (the picker source) AND the server-side state-validation helper SHALL both filter `WHERE synthetic = 0`, or a placeholder id could be offered, pass validation, and drive an `issueUpdate` with a non-real state id → upstream rejection / wrong-state write.
- **Assign-to-me viewer guard (reviews #11, N5)**: `viewer_id` (`mirror.rs:336`) can be unresolved (`None`). An "assign to me" SHALL error explicitly when `viewer_id` is `None` rather than fall through to `assigneeId: null` — which would do the *opposite* of intent (clear the assignee). This guard is **frontend-only by design**: `linear_set_assignee(issue_id, Option<String>)` legitimately accepts `None` for the explicit "unassign" action, so the backend cannot distinguish "unassign" from "assign-to-me with an unresolved viewer." Because assignee writes are intentionally NOT token-gated (the reversible-write carve-out, Decision 5), a renderer-only guard here is consistent with that accepted-risk boundary — not the structural gate the closure/comment writes get. Named explicitly so it is not mistaken for a backend invariant.
- **Assignee don't-touch vs set-null (review N4)**: `IssueUpdateInput.assignee_id: Option<Option<String>>` distinguishes omit / null / set, but the outer `Option` needs `#[serde(skip_serializing_if = "Option::is_none")]` or serde emits `assigneeId: null` for a state-only write — silently unassigning. A test SHALL assert a state-only `IssueUpdateInput` **omits** the `assigneeId` key entirely (not `assigneeId: null`).

## Risks

- **[Risk] Echo filtering before the assignment notification** (the Linear-specific sharp edge): the assignee echo filter MUST run in the run loop on `outcome.newly_assigned` **before** `notify_assignments` (`mod.rs:1033`), NOT inside the pure `reconcile` (review #6 — the registry/AppHandle aren't in `reconcile`'s scope). Pinned with a regression test (own assignment-write does not self-notify), since `notify_assignments` lives in `linear-mirror`'s run loop and a future edit could silently break the order.
- **[Risk] TOCTOU between API-ack and registry record** (review #7): a routine write that records *after* the `await` would leave a window where a concurrent poll sees the new server value with no registry entry → spurious `linear:assigned` / conflict warning. Mitigated by recording a **provisional** entry before the call and clearing it on failure (Decision 2). The residual crash-loss window (process dies after the write lands, before the next poll) collapses to the benign one-shot below.
- **[Risk] Partial-closure irreversibility**: state + comment are independent writes; if the comment posts but the state write fails, the comment cannot be un-posted. Do NOT present per-write rollback as clean — surface "comment posted; state change failed — retry state?" and let the user reconcile. The comment (Decision 4) is never optimistically rolled back. The local close-out (unbind + marker) is applied regardless, since it is the core act.
- **[Risk] Recent-writes TTL vs poll**: entries expire ≥ ~2× the poll interval (derived from cadence, not a magic constant) so the echo poll always arrives before expiry, and a silently-failed write's stale entry never suppresses a real remote change for an unbounded window.
- **[Risk] Closure comment prefill provenance**: user-authored by default; the ship-triggered prefill is the PR link (Nergal-generated, trusted). If ever seeded from agent output it is clearly marked editable and **sanitized** before posting (neutralize Linear `@`-mention syntax so an injected summary can't ping the team). The user always reviews it (Decision 5 confirm) before it posts.
- **[Risk, benign] `recent_writes` crash-loss**: if the daemon crashes after a successful write but before the echo poll, the in-memory entry is gone and the next poll attributes the user's own edit to a remote change → one spurious "remote change" toast (and, for assignee, one spurious `linear:assigned`). No data is wrong. Benign one-shot; does not justify a durable table. Noted so a tester doesn't file it as a bug.
- **[Risk, accepted] Value-equality echo masks a coincidental same-value remote write**: if a remote independently sets the same state and the user's write actually failed, the failure is masked. Both states reconcile to the user-intended value; low blast radius for a single-user tool. Accepted tradeoff of value-comparison echo.
- **[Risk, accepted] Routine reversible writes are not token-gated**: only the closure + comments carry the confirmation token; routine state/assignee edits rely on command-boundary validation + sanitized rendering (`linear-mirror`'s `MarkdownView` already sanitizes). A renderer bug could fire a *reversible* change against the full-access key — a conscious boundary, flagged for the build-time security reviewer.
- **[Risk, cosmetic] Comment echo-match collapse**: two genuinely-identical user comments (same author+body+~timestamp) could dedup one away in the render; rare, recoverable on the next full detail fetch.

## Revision 1 — Detail-modal walk-fixes + project-select (user walk 2026-06-18)

The first dev walk of the write controls surfaced fixes; bundled here (the change is not yet committed/archived). Most are the write controls' own UX; a few are read-fidelity bugs in the detail modal the write controls sit inside; one is a panel feature the user requested in the same pass.

**Write-control UX (the core of this revision):**
- **R1.1 Inline comment composer (item 1)**: the writeback comment composer was a toolbar popup; replace with an always-visible inline `<textarea>` in the comments stream (mirror `ClickUpTaskView.tsx:1153`), carrying `data-nav-key="comment"` so keyboard nav reaches it (Enter focuses it; Esc returns to the nav container). Still routes through `request_comment_token` → `execute_gated_write` (close_out=false) — the security boundary is unchanged.
- **R1.2 State control in the Properties rail (item 2)**: the rail showed a read-only icon+name; the picker was a toolbar icon. Replace with a keyboard-reachable state pill (`data-nav-key="status"`, full state label + glyph, not just a dot) mirroring ClickUp's `StatusPicker`; Enter opens the dropdown (own ↑/↓/Enter/Esc capture), the dropdown writes via `linear_set_issue_state` with the optimistic overlay.
- **R1.3 Keyboard nav over inputs + smooth scroll (item 1/F)**: the nav graph (`navKeysInOrder`) now includes the composer; the `navKey` scroll effect adds ClickUp's special case (status/due land at rail-top → scroll the main area to top; everything else `scrollIntoView({behavior:'smooth'})`).
- **R1.4 Comment appears immediately (item 8)**: after a successful post, the open detail did not refresh (it renders live-fetched comments). Re-fetch `issue_detail` (or append the posted comment to the detail state) on post-success so it shows without reopening.
- **R1.5 Close-out closes the detail modal (item 4)**: `finishCloseOut` closed only the `LinearClosureDialog`; it must also set `linearDetailIssueIdAtom=null` so the detail modal closes when the close-out cycle ends.
- **R1.6 Un-close-out (item 3)**: there was no way to revert a worked-closed marker. Add `mirror::unmark_closed_out` + command `linear_unmark_closed_out` (DELETE FROM linear_closed_out) + a frontend "un-close-out" affordance on closed-out issues (clears the badge; the Linear state, if it was moved, is reverted separately via the now-functional state picker).

**Read-fidelity bugs in the detail modal (pre-existing, fixed in the same pass):**
- **R1.7 Timestamp "Jan 21 1970" (item 7) — root cause**: `model::iso8601_to_epoch` returns epoch **seconds** (`model.rs:314`); the frontend passes them straight to `new Date(x)` / `Date.now() - x` which expect **milliseconds** (`LinearTaskView.tsx:87,92,620`). 1.7e9 s read as ms ≈ Jan 1970. Fix: `*1000` at every consumer (Updated, Activity relative time, comment dates, due-date compare). Frontend-only.
- **R1.8 Comment threads (item 9)**: replies render flat because the GraphQL `ISSUE_DETAIL_QUERY` never fetches `parent { id }` (`client.rs:722`) and neither `LinearComment` (model + mod.rs mapping + `stores/linear.ts`) carries a parent id. Add `parent { id }` to the query + `parent_id` through the struct/mapping/type + nest replies under their parent in the render (mirror Linear's nesting, Image #3).
- **R1.9 T-shirt estimate (item 6)**: the team's `estimationType` (notUsed/exponential/fibonacci/linear/tShirt) is never fetched, so the detail hardcodes "points". Add `estimationType` to the team query + persist per team (`linear_teams`) + expose it; when the issue's team type is `tShirt`, map the numeric estimate to a label (1=XS, 2=S, 3=M, 4=L, 5=XL, 6=XXL — verified against the user's `estimate=2 → S` screenshot); otherwise keep the number. Root-cause (fetch the type), not a value heuristic.
- **R1.10 Activity feed visual (item 5)**: restyle the Activity section to match ClickUp's detail activity feed (the design in the user's screenshots).

**Panel feature (item 10):**
- **R1.11 Single-project select in the "project" view**: today the "project" chip groups ALL issues by project. Add a project `<Select>` (parallel to the team Select) that filters to ONE project grouped by state (like the "mine" chip groups by state), plus an "Todos" option = the current group-all-projects behavior. Needs a `linear_read_projects` command (the `linear_projects` mirror table has no read command) + a `linearProjectFilterAtom` + a filter pass in the panel pipeline.
