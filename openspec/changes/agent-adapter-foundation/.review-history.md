# Independent Plan Review — agent-adapter-foundation

1. **[CRITICAL] Cost extraction trait API mismatches existing implementation; the per-line "RawCost" model is a new behavior, not a refactor.** Task 2.3 says "remove hardcoded Sonnet 4 pricing" and "return `RawCost` only", and the trait sketch defines `extract_cost(line: &str) -> Option<RawCost>` (per-line). But `claude/cost.rs` (verified: lines 48–89) operates on the **whole transcript file** with accumulation logic (`summary.input_tokens += v; …`). The plan does not specify where the accumulation now lives, who owns the running totals per session, or how partial-line streaming reads will reconcile with per-line emissions. Task 11.4 only tests "RawCost on a fixture transcript line" — the integration with the existing aggregator that feeds `CostSummary` to the StatusBar is undefined. This is a load-bearing redesign, not a re-export.

2. **[CRITICAL] No transitional re-exports in tasks; the migration path described in design.md is not actually broken into commits that build.** The design's "Migration path" step 2 says "old `claude/*` modules re-export from `agents/claude_code/*` (transitional)", but task 2.11 deletes `src-tauri/src/claude/` outright with no preceding step adding the re-exports. Verified import sites: `db.rs:8`, `commands.rs:5`, `plan_state.rs:5` all reference `crate::claude::*`. If 2.11 lands without prior re-export shims, the build breaks mid-PR. Either (a) split into commits with shims, or (b) update every importer in the same change and drop the "transitional" claim from design.md.

3. **[CRITICAL] Hook event → `agent_id` resolution introduces a race with session creation and an unindexed-lookup hot path.** Task 3.1: "resolve `agent_id` from `cluihud_session_id` (lookup in session store) and delegate event parsing/dispatch". A `SessionStart` hook can fire before the session row is committed (PTY spawn writes the row, but the hook executes inside the spawned process — ordering is not guaranteed). Plan does not specify: (a) what happens when lookup returns `None` (drop? buffer? assume CC?); (b) whether the lookup hits SQLite per event (potentially thousands of events/session); (c) whether an in-memory cache exists. Currently `hooks/server.rs:226–227` already drops events without `cluihud_session_id` — the plan's resolution step adds a second drop point with no spec for it.

4. **[HIGH] `AgentId` as a string newtype has no validation, but is consumed by DB writes, env vars, and (likely) filesystem paths.** Decision 3 chooses `String` for plugin-extensibility but does not define a charset, length cap, or normalization. Plugins could register `"../../etc/passwd"` or `""`. Used in: `sessions.agent_id` (DB), `set_agent_override` (config persistence), session-creation Tauri command. Spec must define: allowed charset (e.g. `^[a-z][a-z0-9-]{1,31}$`), validation point (constructor returns `Result`), and the registry's behavior on duplicate IDs.

5. **[HIGH] Missing types in task 1.3.** The trait sketch references `EventSink`, `PlanDecision`, `AdapterError` (with `NotSupported`, `SessionLocked` variants), and the spec uses them in scenarios — but task 1.3 enumerates only `DetectionResult`, `SpawnContext`, `SpawnSpec`, `TranscriptEvent`, `RawCost`. Add `EventSink`, `PlanDecision`, `AdapterError`, and the error variants explicitly, otherwise the trait doesn't compile.

6. **[HIGH] Cost USD removal is a user-visible regression with an unbounded timeline.** Decision 6 + Risk R4 + cc-adapter spec ("Status bar shows tokens, not USD, post-foundation") explicitly remove USD display and defer the replacement to "a separate `pricing` module" with no scope, owner, or change ID. The CLAUDE.md framing of "What cluihud is NOT" emphasizes UX-improving changes; downgrading the StatusBar to raw tokens until a future change lands is a UX regression. Either (a) keep CC's USD calculation inside `agents/claude_code/cost.rs` as adapter-private metadata until the pricing module ships, or (b) commit to landing the pricing module in the same release.

7. **[HIGH] DB migration has no rollback path and no test for the backfill.** Task 5.1 adds `agent_id TEXT NOT NULL DEFAULT 'claude-code'`. SQLite's `ALTER TABLE … ADD COLUMN` with `NOT NULL DEFAULT` succeeds in modern versions, but: (a) no test in task 11 verifies the backfill on a populated pre-foundation DB; (b) downgrade story is undefined — SQLite cannot drop columns prior to 3.35; (c) if the migration runs but a later step fails, the user is on a half-migrated DB. Add 11.7 covering pre-existing rows, and document the downgrade path (or accept "no downgrade" explicitly).

8. **[HIGH] `agent_overrides: HashMap<PathBuf, AgentId>` has key-canonicalization bugs.** Task 6.1. `/foo/bar` vs `/foo/bar/` vs `/foo/./bar` vs symlinked paths all hash differently. A user setting an override on `/home/user/Projects/x` won't match a session opened via `/home/user/Projects/./x`. Either canonicalize on write+read, or store as a normalized `String`. Also: `PathBuf` JSON-serialized as map keys is technically platform-specific and serde requires custom handling — verify it actually works in the existing config schema.

9. **[HIGH] Capability gating in stores has a TOCTOU window on session activation.** Task 7.2 populates `agentCapabilitiesAtom` "on session activation, via `invoke<AgentMetadata>("get_session_agent")`" — an async round-trip. Task 7.3 then drops events whose capability flag isn't set. Between `setupHookListeners()` firing and the metadata invoke resolving, the capability set is empty, so legitimate CC events get silently dropped (and only logged to dev console per the spec's defense-in-depth scenario). The spec contradicts itself: "this should be impossible by construction, but treat as defense-in-depth" — but the construction is racy. Either (a) populate capabilities synchronously from session row (cache the capability set on session row creation), or (b) buffer events until capabilities load.

10. **[MEDIUM] `setup_agent` for non-CC throws `NotImplemented` but Settings UI exposes the button.** Tasks 3.3 + 10.1: NotImplemented is thrown for non-CC, yet the Settings panel renders a "Run setup" button per detected agent. Pressing it surfaces an error to the user. Either gate the button on `adapter.has_setup_implementation()` or have non-CC adapters return a "no setup needed" success.

11. **[MEDIUM] `rescan-agents` CLI subcommand is in the spec but not the tasks.** The agent-adapter spec scenario "User installs an agent post-startup" mentions `cluihud rescan-agents`, but task 1.6 only adds `list_available_agents` Tauri command — not a CLI subcommand. Currently `hooks/cli.rs` only has hook subcommands. Missing task: extend `clap` to support a top-level `rescan-agents` subcommand and wire it (probably via Unix socket?) to the running app.

12. **[MEDIUM] `claude --version` during `detect()` blocks startup.** cc-adapter spec Scenario "CC detected with binary and config" returns "version: <claude --version output if cheap to call>". Spawning a child process at app startup adds 50–200ms per detected agent on cold cache. With 4 adapters, this is 200–800ms of blocking before the UI shows. Either defer version capture to first-use or run `detect()` on a background tokio task and update the atom asynchronously.

13. **[MEDIUM] `bitflags` version, frontend serialization shape, and TS atom typing inconsistency are unspecified.** (a) `bitflags::bitflags!` syntax differs between v1.x and v2.x; the workspace `Cargo.toml` is not pinned in the plan. (b) The "wire format `Vec<&'static str>`" requires a custom `Serialize` impl on the bitflags struct — not in tasks. (c) `agentCapabilitiesAtom: Set<AgentCapability>` (design.md) vs `Set<string>` (spec). Pick one and ensure type unification across `src/stores/agent.ts` and the IPC boundary.

14. **[MEDIUM] `parse_transcript_line` on a hot path through `Arc<dyn AgentAdapter>` adds dynamic dispatch overhead.** The current `parse_cost_from_transcript` runs at PTY tail rate; routing every line through `dyn AgentAdapter` works but the plan should call out: hot-path methods stay sync, the dispatcher must avoid re-locking the registry per line, and benchmarks should compare pre/post if any noticeable change is suspected. No benchmark task exists.

15. **[MEDIUM] Plan watcher path policy ignores user config.** cc-adapter spec Requirement: "Plan watcher … SHALL watch `<cwd>/.claude/plans/` first, falling back to `~/.claude/plans/`". But MEMORY notes `plansDirectory: ".claude/plans"` is configured globally in `~/.claude/settings.json`. If the user customizes `plansDirectory`, the adapter's hardcoded path is wrong. The spec needs to read CC's settings.json `plansDirectory` field and respect it.

16. **[MEDIUM] Zero-regression validation is mostly manual.** Task 11 lists 5 integration tests + a manual walk (11.5). Pre-foundation flows are genuinely numerous (plan FIFO, ask-user FIFO, AnnotatableMarkdownView state machine, transcript watcher debounce, plan watcher fallback, file-changed events, openspec watcher). Manual walks are unreliable for catching regressions in a 13-subsystem refactor — at minimum add tests for: openspec event emission post-move, plan FIFO write/read round-trip, annotation injection via UserPromptSubmit, session resume via `--continue`/`--resume`. Risk R1 acknowledges this but tasks don't cover it.

17. **[MEDIUM] Default agent selection at session creation has no policy when CC is uninstalled.** Task 9.1: "default = CC if available, else first detected". "First detected" depends on registry registration order, which is opaque to the user. Define a stable priority (e.g. CC > Codex > OpenCode > Pi) or surface a config setting `default_agent`.

18. **[LOW] Spec scenarios contradict the actual file movement.** agent-adapter spec says "Plan watcher and plansDirectory remain CC-scoped" with paths inside `agents/claude_code/`, but task 2.5 keeps `openspec.rs` at top-level `src-tauri/src/openspec.rs` while CLAUDE.md currently lists it under `claude/openspec.rs`. Several spec scenarios reference files at paths task 2.x will move — verify all `claude/<x>.rs` references in the spec are updated post-move.

19. **[LOW] `Transport::HttpSse` schema designed now is locked in for OpenCode, but no port-conflict / auth story exists.** The variant's payload includes `base_url: Url`, but OpenCode's `:4096` may collide with another local service and an unauthenticated SSE channel could be subscribed to by any local process. Foundation locks the type signature; if auth (token/header) is needed later, the variant's shape changes, breaking other adapters' code that pattern-matches `Transport`. Either add `auth: Option<AuthScheme>` now or document the limitation explicitly.

20. **[LOW] `CLUIHUD_SESSION_ID` injection is mandated in the trait spec but enforcement is implementation-trust.** agent-adapter spec Requirement says `SpawnSpec.env SHALL include CLUIHUD_SESSION_ID`. There's no test verifying every adapter complies. Add a generic test (in the registry tests at task 1.7) that calls `spawn(ctx)` on every registered adapter and asserts `CLUIHUD_SESSION_ID` is present.

21. **[LOW] No mention of `tasks/transcript_parser.rs` move.** cc-adapter spec Requirement "Task parser remains CC tool-name-aware" references `agents/claude_code/tasks.rs (moved from tasks/transcript_parser.rs)`, but the move isn't in tasks 2.x. Add a 2.12 covering it.

22. **[LOW] Frontend session metadata fetch uses a new Tauri command not enumerated.** Task 7.2 invokes `get_session_agent` but no task creates this command; Task 1.6 only adds `list_available_agents`. Add to commands list.

## Round 1 issue resolution audit

All 22 Round 1 issues were addressed substantively. Spot-checks:

- **#1 (cost API mismatch)**: Resolved via `SessionCostAggregator` in `agents/cost_aggregator.rs` (Decision 6, tasks 1.9 + 11.4 + 11.5). Aggregation ownership now explicit.
- **#2 (no transitional re-exports)**: Resolved by dropping the re-export claim entirely; tasks 2.1–2.6 are now atomic in a single commit with explicit grep verification (2.6, 2.15).
- **#3 (hook → agent_id race + unindexed lookup)**: Resolved by `DashMap` cache populated **before** PTY spawn (Decision 9, task 3.1) with DB fallback and drop-on-miss; tests 11.11, 11.12 cover.
- **#4 (AgentId validation)**: Regex + `AdapterError::InvalidAgentId` + duplicate rejection in registry; test 1.12.
- **#5 (missing types)**: All enumerated in trait sketch and tasks 1.4–1.6.
- **#6 (USD regression)**: `legacy_usd_for_sonnet4` private to CC adapter + `get_session_cost_usd` Tauri command preserves UX (task 8.4 + cc-adapter spec).
- **#7 (DB migration test/rollback)**: Test 11.7 + explicit "no downgrade" doc (task 5.4 + R3).
- **#8 (path canonicalization)**: `dunce::canonicalize`, stored as String, test 6.6.
- **#9 (capability TOCTOU)**: Decision 7 makes capabilities sync via session-row payload (task 7.1).
- **#13 (bitflags pinning + serialization)**: `bitflags = "2"` pinned in 1.1; custom Serialize as `Vec<&'static str>` in 1.7; frontend uses `Set<string>` exclusively in 7.2.
- **#16 (zero-regression tests)**: Tasks 11.1–11.13 now cover plan flow, ask-user, tasks, cost, openspec, annotation injection, session resume, file-changed, orphan events, cache miss/recovery.

## Remaining issues (all non-blocking, but must be fixed in-flight)

1. **[HIGH] design.md Decision 7 self-contradicts.** The first paragraph says "elegimos B por simplicidad", and the next paragraph says "**Elegimos (A): simple, source-of-truth única (la session row), zero races.**" Tasks 7.1 implement (A), so the (B) sentence is a stale draft. Fix: delete the trailing "; elegimos B por simplicidad" clause from the bullet list intro. A reader who stops at the first decision will implement the wrong approach.

2. **[MEDIUM] Transcript watcher and plan watcher startup point unspecified post-refactor.** Task 2.12 says `ClaudeCodeAdapter::start_event_pump` is a noop because events arrive via the shared Unix socket. But the existing transcript watcher (`agents/claude_code/transcript.rs`) and plan watcher (`agents/claude_code/plan.rs`) need to be started somewhere on session activation. Pre-refactor they live in `claude/`'s session-lifecycle wiring; post-refactor that wiring is unmoored. Spec should say either (a) `start_event_pump` does start the file watchers for CC (despite the socket being shared), or (b) the watchers stay outside the adapter and the existing wiring continues unchanged. Without this clarification, an implementer following task 2.12 verbatim could ship a no-watcher CC adapter.

3. **[MEDIUM] OpenSpec watcher lifecycle post-move not specified.** Decision 8 + task 2.4 move `openspec.rs` to top-level (correct). But who registers its watcher? Pre-foundation it presumably hooked off CC session lifecycle. Post-foundation it must be project-scoped (any agent in a project with `openspec/`). The plan needs a sentence explicitly saying "openspec watcher is registered per-workspace, not per-session, on workspace open." Test 11.6 verifies behavior but not registration timing.

4. **[MEDIUM] `cluihud rescan-agents` CLI → running app needs a message-type discriminator on the Unix socket.** Task 1.11 says "routes via Unix socket to the running app's `rescan_agents` handler", but the socket today carries `cluihud hook send <event>` payloads. Mixing control commands (`rescan-agents`) onto the same socket needs a discriminator field in the wire schema (`{"kind": "control", "op": "rescan_agents"}` vs `{"kind": "hook_event", ...}`). Spec should call this out; otherwise the dispatcher will try to parse a control message as a hook event and the orphan-drop path eats it.

5. **[LOW] AgentId regex inconsistency between prose and implementation.** Decision 3 prose says regex `^[a-z][a-z0-9-]{1,31}$` (i.e., 2-32 chars). Trait sketch + task 1.2 implement `^[a-z][a-z0-9-]{0,31}$` (1-32 chars, single-letter IDs allowed). The implementation matches the "1-32 chars" prose intent — fix Decision 3's regex string to `{0,31}` for consistency.

6. **[LOW] `AgentId::claude_code()` etc. construct via `Self("claude-code".into())` bypassing validation.** Compile-time correct today, but a future rename via search-and-replace could violate the regex without test failure. Add a `#[test]` that calls each known constructor and asserts `AgentId::new(known.as_str()).is_ok()`.

7. **[LOW] `agent_internal_session_id` column added now but unused in this foundation.** Forward-compat for Pi/Codex resume is reasonable (avoids second migration), but the spec should explicitly mark this as "reserved for future use, NULL for CC" so a reviewer doesn't try to populate it. Task 5.2 should note CC writes NULL.

8. **[LOW] `set_agent_override(project_path, agent_id)` Tauri command takes arbitrary path.** Desktop app threat model makes this acceptable, but the canonicalize-on-write path doesn't reject paths that don't exist (task 6.3 says it falls back to `to_string_lossy`). A typo in the project path silently creates a dead override. Either reject non-existent paths or document the deferred-canonicalization quirk in the spec.

9. **[LOW] Decision 11 (`Transport::HttpSse.auth`) extension story unclear.** Adding a new `AuthScheme` variant later (e.g., `OAuth2 {...}`) is non-breaking only if downstream `match` blocks use `_ =>`. Spec should require non-exhaustive matching or mark `AuthScheme` `#[non_exhaustive]` to prevent the foundation from locking in an over-restrictive contract.

## Verdict rationale

The 3 CRITICAL and 6 HIGH-priority Round 1 issues are all addressed with concrete code-level resolutions, not hand-waving. The migration path is now atomic with explicit `cargo check` checkpoints. The USD regression is preserved via a private bridge — a clean compromise that avoids both "trait pollution with USD" and "user UX regression". Capability gating is sync via session-row payload, eliminating the TOCTOU window. DB migration has test coverage on populated rows.

The remaining issues are polish-level: one design.md self-contradiction (one-line fix), three watcher/socket lifecycle clarifications that an experienced implementer will resolve correctly by reading the existing code, and four low-severity hardening notes. None of them block landing the foundation.

VERDICT: APPROVED
