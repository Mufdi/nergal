### Decisions + tradeoffs
- **DB-as-queue, not an in-memory `HashMap<SessionId, PendingDelivery>`** (design §2 named the HashMap as one option). A message is pending iff `agent_consumed_at IS NULL` — the DB is the single source of truth, survives restart, and removes a sync class of bugs. The `SessionDelivery` trait therefore carries only `wake_idle` + `emit`, not `queue_for_stop`/`drain_on_idle` as methods; the queue is implicit and the drain is the free fn `delivery::drain_idle`.
- **Config injected into `send_to_session`** (not `Config::load()` internally) so the security-spine tests run against a literal `CrossSessionConfig` and never mutate the real on-disk config (`config_path()` has no test override).
- **TOCTOU/liveness without a shared lock**: send does insert→re-read-mode→wake-if-idle; the mode-map writer does set-idle→read-DB→wake. DashMap per-key ordering guarantees one of the two always observes the other's effect, so a message is never stranded. A benign double-wake is possible (idempotent: `read_messages` drains all pending).

### Divergencias vs proposal/tasks
- **Task 4.3 (`additionalContext` Stop-CLI fast-path) DEFERRED — documented, not silently skipped.** The idle-transition PTY drain (task 4.4) is the load-bearing, universal delivery mechanism and FULLY satisfies every "criterio de done" (idle, working, and send-just-after-Stop with no stranding — see below). The `additionalContext` path only changes *when* delivery lands (same turn vs next idle), never *whether*. Implementing it naively conflicts with the idle-drain: cluihud sees the same `Stop` event and cannot tell whether CC continued the turn (additionalContext emitted) or idled, so running both risks a PTY wake landing mid-turn (corruption) — the exact invariant the design forbids. The clean coexistence requires the stop-CLI to coordinate a "keep-running" signal with the daemon over a new FIFO round-trip on every Stop (a hot path), adding latency + a failure mode for a latency optimization the design itself frames as "best-effort … never instead of the idle-transition drain". Deferred as a follow-up; surfaced at review. Mirrors how change-1 deferred phase-6 initially.
  - **Done-criteria coverage WITHOUT 4.3**: idle target → immediate wake (send path); working target → queued, delivered on next `Stop` via the idle-drain; send-just-after-Stop → either send sees idle and wakes, or the in-flight Stop's idle-drain wakes. No stranding in any branch.
- **`search_sessions` is name+summary substring over the live+recent roster** (reuses `directory::list_sessions`), not ripgrep transcript full-text (impl §3 mentioned the `search/mod.rs` transcripts scope as an option). The security-relevant requirement — find an inactive session read-only and refuse to message it — is fully met (the refusal lives in `send_to_session`'s active-only check; search is read-only by construction). Transcript full-text is a future enhancement.
- **Thread-close notification = frontend event only** (`crossmsg:thread-closed`), not a PTY wake to participants. Waking live PTYs to announce a closure is noise mid-work; the panel surfaces the closed thread (the design's "surface a stuck thread to the user" intent). The deadline sweeper drives it.

### Security review (critical-tier escalation — security-auditor)
Verdict went DO-NOT-SHIP → **SHIP** after fixes. Applied:
- **HIGH — C1 control bypass.** `delivery::sanitize_for_pty` used `!c.is_control()`, which misses the 8-bit C1 range (U+0080..=U+009F); a relayed `\u{009b}201~` (8-bit CSI) could close the bracketed paste on a C1-honoring terminal. Fixed by delegating to the single canonical `crate::pty::sanitize_for_pty` (consumes ESC-CSI/OSC sequences AND strips C1) — also resolves the reviewer's "two divergent sanitizers" maintenance flag.
- **MEDIUM — dedup hash not stable across restarts.** `DefaultHasher` (SipHash) is seed-randomized per process, so a stored `dedup_key` never matched after a restart → identical message re-delivered. Switched to SHA-256 (deterministic).
- **MEDIUM — non-participant could post to a borrowed thread_id** at reach level 0. Added a participant gate in `send_to_session`: only an existing participant (a session pulled in as a recipient) may post; self-joining with a borrowed id returns `not_a_participant`. New test `non_participant_cannot_post_to_borrowed_thread`.
- **LOW — `now_secs()` clock-zero.** Pre-epoch clock returned 0, making the sweeper close every active thread on recovery. Now returns a far-future sentinel.
- Accepted/deferred LOW: dedup check+insert TOCTOU (bounded by msg_budget + per-connection serialization), O(N) `threads_for` scan (bounded by defaults; total-thread cap is a future enhancement), indirect prompt-injection via relayed AI summary (inherent to any agent↔agent relay; mitigated by the advisory labeling). Confirmed clean: SQLi (all `params![]`), kill-switch gates every delivery path, `human_seen_at` never cancels `agent_consumed_at`, aux-shell write guard intact, no triggerable unwrap/panic.

### Walk round 1 fixes (post first live walk)
- **Idle target stranded (queued, never woken).** The send-path wake required `mode == "idle"` exactly, but the runtime mode side-map is volatile (empty after restart / for a session quiet since the daemon started) → a quiet idle target queued forever (no `Stop` to trigger the idle-drain). Fixed: wake unless the target is `running`/`needs_attention`; idle/completed/unknown are woken (a paste into a momentarily-busy REPL is buffered, not corrupting). New tests `unknown_mode_target_is_woken_not_stranded`, `idle_target_delivery_embeds_and_consumes`.
- **Panel: entering a thread did nothing.** `CrossSessionPanel` used `getDefaultStore()`, but the app mounts its Provider with a custom `appStore` (`jotaiStore.ts`) → clicks wrote `activeThreadId` to the wrong store. Fixed to `useStore()` (context store).
- **Latency (agility ask).** The wake note now EMBEDS the sanitized message bodies + origin + thread id, and delivery consumes (`agent_consumed_at` set on a successful wake) — the receiving agent reads+replies in one turn with no `read_messages` round-trip. `read_messages` stays as a catch-up/full-history fallback (still take-on-read for the queued/missed-wake path). **Semantic note**: for the wake path, delivery == consume (was take-on-read); consistent with the already-accepted at-most-once posture (deadline sweeper + human panel are the net). Update the spec delta accordingly at `/openspec-sync`.
- **Live UX.** On `crossmsg:new` the right panel SOFT-opens to the live thread (expand + select, streams in) — only when collapsed or already on Cross-session, never overriding a panel the user chose, never stealing focus, never locking. Auto-open selects without marking seen; the unread badge clears only when the user engages the panel (`onMouseDown`/`onFocus`).
- **Unread indicator** on `SessionRow` changed from a plain dot (confusable with the status dots) to a chat-bubble glyph + count in an accent pill.

### Walk round 2 fixes
- **Indicator on BOTH participants of a live thread.** The badge keyed only on human-unread (recipient-only), so a thread showed on the recipient but not the sender. Added `crossSessionActiveParticipantsAtom` (session ids in any `status=="active"` thread); `SessionRow` shows the chat glyph for any active-thread participant, with the unread count overlaid when the human hasn't looked (accent vs muted styling).
