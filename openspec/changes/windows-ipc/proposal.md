## Why

`windows-compile` (change #1) makes the crate build on Windows but leaves the hook server, MCP daemon, cross-session messaging, and the plan-review / ask-user blocking round-trips as `Unsupported` stubs. Those ARE Nergal's core loop — observing and augmenting the agent. This change implements the real **Windows named-pipe transport** so they work, fulfilling the `platform-ipc-transport` spec's already-written **deferred Windows requirements** (Windows transport contract, Windows peer-authentication, blocking-primitive-without-mkfifo) with the **same owner-only security boundary** the Unix `SO_PEERCRED` uid wall provides today.

The transport seam (`platform::PlatformListener`/`PlatformStream`) already exists with `#[cfg(windows)]` stubs. This change replaces the stub bodies with a `tokio::net::windows::named_pipe` implementation and migrates the hook server + MCP daemon + blocking round-trips onto the seam (on all platforms; the Unix body is unchanged, so no Linux/macOS behaviour change beyond what the macOS iteration already shipped).

Security posture, grounded in verified Win32/tokio APIs (`handoff/windows-namedpipe-research.md`): the named pipe is the spec's **recommended** transport (Decision 3 of the archived `platform-ipc`) because it maps the Unix model directly — an OS-enforced owner-only ACL (the security-descriptor analog of `0600` + the per-user `0700` dir) plus a real client-SID peer identity (the analog of the peer-cred uid). The loopback+token fallback stays **spec-only / not implemented** here: named-pipe has no edge case forcing it, and it trades an OS-enforced ACL for an in-band secret. **The user has a Windows machine to walk this** — the security boundary is runtime-validated, not compile-only.

## What Changes

- **Named-pipe `PlatformListener`/`PlatformStream` bodies** — replace the `#[cfg(windows)]` `Unsupported` stubs (`platform/mod.rs:730-807`) with `tokio::net::windows::named_pipe` `ServerOptions`/`ClientOptions`. `NamedPipeServer`/`NamedPipeClient` already implement `AsyncRead + AsyncWrite`, so the length-framing helpers (`read_frame`/`write_frame`) operate unchanged over the abstracted stream.
- **Per-user pipe naming + anti-squat** — endpoints become `\\.\pipe\nergal-<user-SID-string>-<endpoint>` (per-user isolation; the SID is unique per user, collision-free on a multi-user box). The server binds with `first_pipe_instance(true)` so a pre-created squatter pipe makes `create()` fail (`PermissionDenied`) → the server refuses to start (security alert) rather than binding a hostile name. The **client** verifies the connected pipe's owner SID via `GetSecurityInfo(SE_KERNEL_OBJECT, OWNER_SECURITY_INFORMATION)` equals the current user before trusting it (anti-impersonation — closes the different-user-same-name gap).
- **Owner-only security descriptor** — the server pipe is created with a `SECURITY_ATTRIBUTES` built from the SDDL `D:P(A;;GA;;;<user-SID>)` (protected DACL, GENERIC_ALL to the current user only) via `ConvertStringSecurityDescriptorToSecurityDescriptorW`, passed to `create_with_security_attributes_raw`. This is the ACL analog of the Unix `0600` socket inside the `0700` per-user dir.
- **Client-SID peer authentication (defense in depth)** — on each accepted connection the server extracts the client's user SID (`ImpersonateNamedPipeClient` → `OpenThreadToken(openasself=true)` → `GetTokenInformation(TokenUser)` → `RevertToSelf`, RAII-guarded) and rejects it if it does not `EqualSid` the server process's owner SID. Mirrors the Unix peer-cred uid check; logged + rate-limited per the IPC-observability requirement.
- **Peer-identity type refactor** — `PlatformListener::accept` today returns `(PlatformStream, u32)` (a Unix uid). Replace the `u32` with an opaque `PeerIdentity` (`enum`: Unix `uid: u32` / Windows `sid`) exposing `matches_current_process() -> bool`. The hook-server and MCP accept loops compare via that method instead of a raw `u32 != app_uid`. Unix behaviour is byte-identical (the method is `uid == getuid()`).
- **Migrate hook server + MCP daemon onto the seam** — `windows-compile` gated `hooks/server.rs` (raw `UnixListener`) and `mcp/transport.rs` (`UnixSocketTransport`) `#[cfg(unix)]`. This change routes both through `PlatformListener`/`PlatformStream` so they obtain their listener from the abstraction (the spec's "Platform transport abstraction" requirement) and therefore work on Windows. The MCP `UnixSocketTransport` is replaced by `PlatformListener` + the unchanged framing helpers.
- **Migrate plan-review / ask-user onto `PlatformStream`** — replace the broken `#[cfg(not(unix))]` shell-`mkfifo` placeholder (`hooks/cli.rs:197-207`, which today unlinks its own FIFO before the round-trip even runs). Mirroring the Unix CLI-owns-endpoint model, the hook **CLI hosts a synchronous named-pipe server** for the gate (a new `#[cfg(windows)]` sync-server primitive — the CLI is a blocking binary with no tokio runtime); the **GUI connects when the plan-review notification arrives and holds the connection open** during deliberation, writing the decision over it when the user submits (`submit_plan_decision` writes over the held `PlatformStream` on Windows, not `std::fs::write` against a pipe name). Because the connection is held end-to-end, GUI death surfaces as a clean connection EOF → safe deny (stronger than the Unix FIFO's pid+start-time poll), while preserving block-until-decision + the owner-only boundary (server SD + accept-side client-SID check, so a forged allow is impossible).
- **Cross-session messaging** rides the MCP daemon socket (no separate listener — it is DB-as-queue dispatched in `mcp/mod.rs`), so migrating the MCP daemon onto the seam covers it automatically.
- **`sync_connect` Windows body** — replace `windows-compile`'s `UnsupportedSyncStream` with a synchronous named-pipe connect for the hook-CLI blocking binary.
- **Windows endpoint-path resolver** — `hook_socket_path`/`mcp_socket_path`/`plan_review_fifo_path`/`gui_pid_path` return pipe-name strings on Windows (no filesystem `ipc_dir`); the per-user SID replaces the per-user `0700` directory as the isolation root.
- **Cargo** — add `[target.'cfg(windows)'.dependencies.windows]` with the Win32 features (`Win32_Foundation`, `Win32_Security`, `Win32_Security_Authorization`, `Win32_System_Memory`, `Win32_System_Pipes`, `Win32_System_Threading`, `Win32_System_IO`, `Win32_Storage_FileSystem`); add `getrandom` only if not already transitive (it almost certainly is).
- **NOT implemented: loopback+token fallback** — recorded in the spec as the constrained fallback; named-pipe is sufficient and safer, so this change ships named-pipe only. A future change adopts loopback only if a real named-pipe edge case appears.

## Capabilities

### New Capabilities

(none)

### Modified Capabilities

- `platform-ipc-transport`: implements the Windows half of the existing contract. **Modifies** the deferred requirements — "Windows transport contract (deferred)", "Per-platform peer-authentication boundary" (the Windows clause), "Unified blocking request/response primitive" (the deferred Windows-target scenario), "Platform transport abstraction" (hook + MCP now actually obtain their listener from the seam), and "Platform-gated socket permissions" (Windows owner-only SD replaces the POSIX mode). The "Constrained Windows loopback fallback" requirement stays as-is (still the documented, un-adopted fallback).

## Impact

- **`src-tauri/src/platform/mod.rs`**: real `#[cfg(windows)]` `PlatformListener`/`PlatformStream` (named pipe); `PeerIdentity` opaque type replacing the `u32` from `accept`; `#[cfg(windows)]` `peer_identity_of` (client-SID extraction); Windows branches of the endpoint-path helpers; `#[cfg(windows)]` `sync_connect` (named-pipe). New `#[cfg(windows)]` security module (SDDL build, SID helpers, owner verification).
- **`src-tauri/src/hooks/server.rs`**: bind + accept loop routed through `PlatformListener`, peer check via `PeerIdentity::matches_current_process()`.
- **`src-tauri/src/mcp/transport.rs` + `mcp/mod.rs`**: `UnixSocketTransport` replaced by `PlatformListener`/`PlatformStream`; framing helpers unchanged; daemon accept loop uses `PeerIdentity`.
- **`src-tauri/src/hooks/cli.rs`**: plan-review/ask-user blocking round-trip over `PlatformStream` on Windows (connected-EOF death signal); the broken `#[cfg(not(unix))]` placeholder removed.
- **`src-tauri/src/lib.rs`**: the MCP daemon spawn (gated `#[cfg(unix)]` by `windows-compile`) un-gated to run through the seam on Windows too.
- **`src-tauri/Cargo.toml`**: `[target.'cfg(windows)'.dependencies.windows]` + features; `getrandom` if needed.
- **Out of scope**: process/port introspection (`windows-proc`), desktop-integration detach/reveal (`windows-desktop`), bundling (`windows-bundle-ci`), the loopback+token fallback (spec-only).
