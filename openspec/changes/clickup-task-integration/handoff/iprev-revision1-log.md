# Adversarial review — Revision 1 (run-state map)

## Findings

1. **[CRITICAL] El edge Running NUNCA llega al dispatcher — para ningún agente.** El plan afirma "Claude Code: registers `UserPromptSubmit` + `Stop` hooks (`adapter.rs:113`) → full Running/Idle edges" y lo mismo para Codex. Falso. `transport()` en `src-tauri/src/agents/claude_code/adapter.rs:103-119` es **metadata descriptiva**, no lo que se instala. Lo que realmente se instala es `src-tauri/src/setup.rs:74-81`: `UserPromptSubmit` → `cluihud hook inject-edits` — y `inject_edits()` (`src-tauri/src/hooks/cli.rs:64-109`) **jamás toca el socket**: lee `HookState`, muta stdout y sale. Codex idéntico: `src-tauri/src/agents/codex/setup.rs:95` mapea `UserPromptSubmit` → `inject-edits`. Lo confirmé contra el `~/.claude/settings.json` vivo de esta máquina: `UserPromptSubmit` corre solo `cluihud-conditional.sh inject-edits`. Consecuencia: el writer Running del plan (dispatcher arm `server.rs:732`) **nunca se ejecuta**; el map lee Idle/absent siempre; cada send entrega inmediato. La revisión reconstruye exactamente la premisa falsificada que vino a reemplazar (guard permanentemente abierto), un nivel más abajo. Fix obligatorio: setup.rs (y codex/setup.rs) deben agregar una entry async `cluihud hook send ...` para `UserPromptSubmit` (o hacer que `inject_edits` también dispare el socket message), los usuarios deben re-correr `cluihud setup`, y ambos archivos entran a "Affected artifacts". Nada de esto está en el plan.

2. **[CRITICAL] Lost-wakeup race entre enqueue y drain.** El plan especifica dos `Mutex` separados (run-state map + queue map) sin atomicidad entre check y enqueue. Interleaving fatal: el command lee Running y suelta el lock → llega `Stop`, setea Idle y drena (queue vacía) → el command inserta en la queue → el send queda varado hasta que *otro* turno completo ocurra (que puede no ocurrir nunca). Con hooks `async: true`, Stop llega con latencia arbitraria, así que esta ventana es real, no teórica. Fix: un solo `Mutex` sobre un struct que contenga ambos (check-Running + enqueue atómico vs set-Idle + pop atómico), o re-check del estado después del insert bajo lock order definido. El plan debe especificarlo; "same pattern as `pending_prompts`" (`pty.rs:76`) no aplica — `pending_prompts` tiene un solo consumidor sin guard condicional.

3. **[HIGH] User interrupt (Esc) deja Running stale — y es el caso común, no el crash.** El hook `Stop` de CC no corre cuando el usuario interrumpe el turno. Resultado: map dice Running, agente está idle, todo send subsiguiente se encola y nunca drena (no habrá Stop hasta que el usuario corra otro turno completo a mano). El open question 1 solo considera crash del agente; el PTY-exit cleanup propuesto **no cubre interrupts** (el PTY sigue vivo). Mitigación mínima: documentarlo + drenar en el próximo Stop cuando llegue; mejor: TTL sobre Running o limpiar Running en el próximo `UserPromptSubmit` sin Stop intermedio.

4. **[HIGH] PTY kill/respawn (restart de sesión, no de app) preserva map y queue stale.** `kill_session_pty` (`pty.rs:929-947`) y el arm `SessionEnd` (`server.rs:586-589`) no purgan nada según el plan. Flujo rutinario: send queda queued → usuario mata/reinicia la sesión → `start_claude_session` re-spawnea con el **mismo session_id** → el próximo Stop drena un bloque compuesto contra el estado viejo de la task dentro de una conversación nueva. El plan solo consideró restart de la app. Cleanup en `kill_session_pty` + `SessionEnd` es required, no fast-follow.

5. **[MEDIUM] El claim de Pi es falso.** El plan dice "the jsonl tail folds Cost events into Stop (`jsonl_tail.rs:161`) → Idle edge only". La línea :161 es un **doc comment desactualizado**; el código real (`jsonl_tail.rs:188-206`) traduce `Cost` → `HookEvent::AgentStatus`, nunca `Stop`. Pi no tiene edge Idle ni Running. El outcome práctico coincide con la degradación aceptada (siempre inmediato), pero un claim "verified at build time" que se verificó contra un comment y no contra el código invalida la confianza en la tabla entera — consistente con que la tabla también está mal para CC y Codex (finding 1).

6. **[MEDIUM] Ordering de eventos no garantizado entre conexiones de socket.** Cada invocación de hook abre su propia `UnixStream` y el server la procesa en un `tokio::spawn` independiente (`server.rs:205-260`). Dos eventos cercanos (Stop seguido de UserPromptSubmit del próximo prompt) pueden procesarse invertidos → el map queda Running tras un turno terminado (send varado) o Idle durante generación (paste mid-turn — la falla exacta que el defer previene). Baja probabilidad por evento, pero con un usuario rápido y hooks async es cuestión de tiempo. Mitigación: al menos documentarlo; idealmente sequence/timestamp en el payload.

7. **[MEDIUM] Mecánica del drain subespecificada.** `terminal_paste` (`pty.rs:994`) es un `#[tauri::command]` que toma `State<'_, PtyManager>`; el dispatcher tendría que resolverlo vía `app.state::<PtyManager>()` o un helper interno tipo `write_session_data` (`pty.rs:823`) — que hoy **no hace bracketed paste**. El plan no dice cuál. Además: (a) no sostener el lock del map/queue a través del write al PTY (un writer wedgeado bloquearía el procesamiento de hooks de todas las sesiones); (b) `write_all` a un PTY muerto retorna error — el plan debe definir drop+log, no retry ni silencio; (c) el `\r` debe ir fuera del bracketed-paste wrapper (dentro de `\x1b[200~...\x1b[201~` un `\r` es texto, no submit).

8. **[MEDIUM] El happy path inmediato también tiene TOCTOU inherente.** El usuario submitea un prompt en la terminal; el evento viaja CC → hook CLI (proceso nuevo) → socket → dispatcher. En esa ventana de decenas-a-cientos de ms, el command lee Idle y pega mid-generación. El plan presenta el guard como invariante duro ("a PTY write must never interrupt a generating agent"); con esta señal es best-effort para *todos* los agentes, no solo los degradados. Debe re-documentarse como tal en el design.md.

9. **[MEDIUM] El drain entrega en un momento que el usuario no controla.** Si el usuario está a mitad de tipear su propio prompt cuando llega el Stop, el paste + `\r` se inyecta dentro de su draft y **submitea el texto spliceado**. El path inmediato existente tiene el mismo riesgo pero el usuario acaba de clickear "send" (contexto fresco); el drain puede dispararse minutos después. Mínimo: documentar; mejor: el drain verifica que no haya input pendiente, o entrega sin `\r`.

10. **[LOW] Observabilidad incompleta.** Logs en enqueue/drain están especificados, pero falta: evento al frontend cuando el send queued se entrega o se descarta (el UI muestra "queued" y nunca se entera del desenlace), log en replace (open question 2), y alguna superficie para que el usuario vea/cancele un send pendiente — crítico dado finding 3, donde "queued" puede colgar para siempre sin ninguna señal.

## Open questions — verdicts

**Q1 (Running staleness / PTY-exit cleanup): REQUIRED-FOR-APPROVAL, y ampliada.** El teardown hook es necesario pero insuficiente: debe cubrir también `kill_session_pty` + `SessionEnd` (finding 4) y el caso interrupt (finding 3), que no implica teardown alguno. Tal como está redactada (solo crash), la mitigación cubre el caso raro y omite el cotidiano.

**Q2 (queue depth replace-vs-FIFO): ACCEPTABLE RESIDUAL.** Replace-with-log es la semántica correcta para un prompt imperativo one-shot; FIFO de N invitaría multi-delivery stale. Solo exige el log de replace + señal al UI (finding 10).

**Q3 (subagent noise): NON-ISSUE VERIFICADO — acceptable residual.** No existe variante `SubagentStop` en `events.rs` (grep en todo `src-tauri/src/`: cero matches), `setup.rs` no la registra, y el `settings.json` instalado tampoco (`SubagentStop: null` verificado). CC emite SubagentStop como hook event *separado* de Stop; al no estar registrado, nunca dispara, y si algún día llegara al socket, el serde tag fallaría y se descartaría con warn (`server.rs:244-257`). El writer no necesita filtro. Caveat: documentar que jamás se mapee SubagentStop → payload con `hook_event_name: "Stop"`. El riesgo real en esa zona es el ordering inter-conexión (finding 6), que el plan no vio.

## Summary

La revisión reemplaza una premisa falsificada (columna `status` nunca escrita) por otra premisa falsificada del mismo tipo (evento `UserPromptSubmit` nunca enviado al socket): el guard resultante queda permanentemente abierto para CC, Codex, Pi y OpenCode — es decir, para el 100% de los agentes — mientras el plan lo presenta como "full edges" para los dos primeros. Sumado al lost-wakeup race del diseño de dos mutexes y a los dos paths de staleness no cubiertos (interrupt, kill/respawn), el mecanismo tal como está especificado no entrega la protección que documenta.
