## 6. Desktop notifications — replace notify-send at all three sites

- [ ] 6.0 **Empirical gate — decides the Linux path** (Decision 3/3a): the three `notify-send` sites exist because `tauri_plugin_notification` `.show()` *failed silently* under WebKitGTK (returned `Ok(())`, displayed nothing). Silent failure is NOT detectable at runtime, so the Linux strategy is decided HERE, by observation: wire one site, `pnpm tauri dev`, trigger a notification, and watch the OS notification center.
  - **If it displays** → plugin is the sole Linux path; proceed to remove `notify-send` at all three sites (6.1 helper, plugin-only body).
  - **If it does NOT display** → keep `notify-send` as the **primary** Linux path (6.1 helper, Linux body = `notify-send` spawn behind `#[cfg(target_os = "linux")]`); the plugin is used only on macOS. Do NOT use the plugin on Linux in this branch (its `Ok` is indistinguishable from silent failure).
  - Record the WebKitGTK version + observed result in the change notes — this is the evidence the branch decision rests on.
- [ ] 6.1 Add a shared helper `crate::notify::send(app: &tauri::AppHandle, title: &str, body: &str)` (pin `&AppHandle`; the three callers each pass a borrow). Body depends on the 6.0 outcome:
  - **macOS** (`#[cfg(target_os = "macos")]`): `app.notification().builder().title(title).body(body).show()`; on `Err` → `tracing::warn!`.
  - **Linux** (`#[cfg(target_os = "linux")]`): if 6.0 = displays → same plugin call; if 6.0 = silent → `notify-send` spawn (the prior behavior), on spawn `Err` → `tracing::warn!`. **No `Ok`/`Err` runtime toggle is used as the silent-failure guard** (it cannot catch an `Ok`-no-display — Decision 3a). An optional best-effort `Err`→`notify-send` retry for *loud* plugin failures may be added on the Linux plugin branch, but it is not the silent-failure mechanism and nothing relies on it.
  - **Other targets** (`#[cfg(not(any(target_os = "linux", target_os = "macos")))]`): explicit `tracing::warn!` no-op stub (Windows is a non-goal this pass). This keeps the helper "born `#[cfg]`-gated with a non-Linux stub" per the CLAUDE.md cross-platform invariant, so a future Windows build's behavior is intentional rather than an accidental empty body.
- [ ] 6.2 In `src-tauri/src/commands.rs`, replace `send_notification` (`#[tauri::command]`, lines ~571-582) with `crate::notify::send(&app, &title, &body)`. Add an `app: tauri::AppHandle` parameter (runtime-injected — no frontend change, Decision 5).
- [ ] 6.3 In `src-tauri/src/linear/mod.rs`, replace the `notify-send` subprocess in `notify_assignments` (lines ~1499-1505) with `crate::notify::send(app, "Linear", &body)` (`app: &AppHandle` is already a parameter — pass it directly). Keep the existing `app.emit("linear:assigned", …)` in-app event.
- [ ] 6.3b In `src-tauri/src/clickup/poller.rs`, replace `TauriEffects::notify` (lines ~922-936) with `crate::notify::send(&self.app, title, body)`. Replace the stale comment block (923-925) that claims `.show()` is "unreliable under WebKitGTK" with the Decision 3a rationale (plugin on macOS; Linux path decided by the 6.0 empirical gate). Keep the `emit_changed`/`emit_assigned` events untouched.
- [ ] 6.4 **macOS notification permission, sequenced before pollers** (Decision / Open Questions): in `setup()` on the main thread, if `app.notification().permission_state()` is `NotDetermined`, call `request_permission()` (use `app.run_on_main_thread(...)` if invoked off-main — macOS may require main thread). **Spawn the ClickUp/Linear pollers only AFTER this resolves**, so a poller's first send never races the not-determined window from a background thread. A denied result flows through the non-fatal warn path (no panic, no error to caller). Verify ordering: pollers do not start before permission resolution. **Tradeoff to acknowledge**: on macOS `request_permission()`'s future resolves only when the user answers the dialog, so gating poller *startup* on it delays the pollers until the prompt is answered/dismissed. Acceptable default (avoids the R2-3 race); if the startup delay is undesirable, the alternative is to resolve permission in `setup()` without gating poller startup and let the first poller `.show()` auto-prompt — document whichever is chosen.
