# Adversarial Review — linear-mirror

Reviewing as skeptic. Severities escalated per instructions. The plan is well-structured and the divergences from ClickUp are mostly identified correctly, but several load-bearing correctness claims rest on unstated invariants or unverified API behavior, and the reconcile model has at least one data-destroying failure mode.

1. **[CRITICAL] A truncated fetch is treated as authoritative → mass false tombstoning.** D5/§4.1/§4.3 define the bounded fetch as "authoritative within scope" and tombstone "in-window mirror issues absent from the fetch." D6/§2.3 also allow the client to "give up after a bounded number of retries" on `RATELIMITED`, and cursor pagination can be interrupted mid-loop. Nothing in the plan distinguishes a *complete* in-scope fetch from a *partially-paginated, then-abandoned* fetch. If the rate-limit cap is hit on page 3 of 8, reconcile sees pages 4–8's issues as "in-window but absent" and tombstones every real, active issue on them. §4.1's "mid-cycle network failure commits nothing" is asserted but the give-up-after-retries path is not classified as such. The authoritative/tombstone gate MUST require a provably-complete pagination (`hasNextPage == false` reached for every team) before any tombstone fires; partial fetches must commit upserts only, never tombstones. No test covers this (see #18).

2. **[CRITICAL] `parent_id` self-FK has no insert-ordering or out-of-scope-parent handling.** `implementation.md` confirms `PRAGMA foreign_keys = ON` is live, and §3.1 declares `parent_id TEXT REFERENCES linear_issues(id) ON DELETE CASCADE`. SQLite checks FKs immediately by default. Within one issues batch, a sub-issue can be inserted before its parent (page order is `updatedAt` desc, not topological) → immediate FK violation. Worse, a sub-issue's `parent` can be **out of the polled scope** (parent updated >30 days ago) → `parent_id` references a row that will never be inserted → permanent FK violation. The placeholder-synthesis path (§4.2, edge-cases note) is explicitly scoped to *non-null `state_id`/`team_id` only* and explicitly excludes "legitimately-null relations" — but a non-null `parent_id` to an unfetched parent is neither. This aborts or corrupts the transaction. Needs: deferred FKs or a two-pass/topo insert, **and** placeholder/null-coalesce for unknown parents.

3. **[HIGH] The entire un-assignment-detectability argument rests on an unverified API assumption.** D4 and the spec ("Un-assignment is detectable within the window") hinge on "reassigning an issue advances its `updatedAt` into the window." The "Verified API facts" block anchors that issues are *orderable* by `updatedAt` — it does **not** verify that an assignee mutation bumps `updatedAt`. This is the load-bearing claim for "wrongly keep un-assigned ones": if assignee changes do not bump `updatedAt`, then after un-assignment the issue leaves the `assignee == viewer` branch and, if outside the window, is never refetched → the mirror shows it `assigned_to = viewer` forever. Anchor this against the API or the design is unsound.

4. **[HIGH] Aged-out, non-viewer-assigned issues are never tombstoned and never GC'd → unbounded mirror growth + permanently stale display.** §4.3: "aged-out issues are simply not refreshed." GC (§4.7) only collects *tombstoned* rows. So a team issue updated 31 days ago is neither in the fetch, nor a tombstone candidate (`updated_at < window_start`), nor GC-eligible. It lingers indefinitely showing its 30-day-old state, and the show-completed toggle (linear-task-panel spec) will surface stale `completed`/`canceled` rows that may have been reopened. There is no eviction path for the dominant case. Needs an explicit age-out eviction distinct from tombstone-GC.

5. **[HIGH] `parent_id ON DELETE CASCADE` actively contradicts the childless-GC invariant.** The spec requires "a tombstoned issue with live sub-issues SHALL be retained." But §3.1 schema cascades child issues on parent delete. The retention rule is enforced only in app logic, and the schema is one buggy GC query away from cascade-wiping live subtrees. The cascade on the *self-FK* should be `ON DELETE RESTRICT` (or no action) so the DB itself enforces the invariant the spec promises.

6. **[HIGH] Rate-limit backoff waits for the wrong reset bucket.** §2.3/D6: "Back off until the **nearer** of `X-RateLimit-Requests-Reset` / `X-RateLimit-Complexity-Reset`." If the *complexity* bucket is exhausted (likely, given the heavy nested query — see #11) but requests-remaining is fine, the nearer reset is the requests bucket, which doesn't help. The client retries before complexity resets → `RATELIMITED` again → burns the bounded retries → poll gives up → feeds #1. Must wait until the reset of the bucket that actually returned exhausted, not the nearer of the two.

7. **[HIGH] Untrusted-content sanitization is scoped to attachment chips and misses inline markdown.** The linear-task-panel security requirement gates *attachment thumbnails* against non-image URLs and lazy-loads them. But descriptions/comments render through `MarkdownView` (implementation.md), and inline markdown — `![](http://attacker/pixel.png)` auto-loads on detail open (tracking pixel / IP + viewer-presence leak / SSRF surface in WebKitGTK), and `[x](javascript:…)` / `data:` link schemes — is not covered by the attachment gate. "Strips raw HTML" does not neutralize markdown-native image auto-load or dangerous link schemes. Sanitization must cover inline image hosts and link-scheme allow-listing (http/https only), with a test.

8. **[HIGH] Account/viewer switch does not reset the baseline → notification storm; schema contradicts design.** §3.1 declares `linear_sync_state` as a single row `CHECK(id=1)`, but D3 describes it as "keyed by org/viewer." If the user swaps the API key to a different Linear account, `baseline_done` stays `1`, so the new account's first sync is **not** silent and every pre-assigned issue fires a notification (the exact failure the baseline gate exists to prevent). A `viewer_id` change must reset `baseline_done` and wipe/retire the prior mirror. Not handled; no test.

9. **[HIGH] The viewer-assigned union is workspace-wide and pollutes the mirror with non-selected-team data.** §2.2 lists `viewer_assigned_issues(after)` with no team argument. Per D4 it returns viewer-assigned issues across **all** teams, including unselected ones. Those issues carry a `team_id` whose team and `workflow_states` were never fetched (states are fetched only per selected team) → forced `state_id`/`team_id` placeholder synthesis on every poll, and undefined panel visibility ("Todos" = selected teams). Either constrain the union to selected teams or define how cross-team assigned issues are stored and shown.

10. **[MEDIUM] The window `now` is not pinned across the server filter and the tombstone-candidate query.** D4 fetches `updatedAt > now - window`; the edge-case note computes candidates as `updated_at >= window_start`. If `now` (fetch start) and `window_start` (commit time) are computed independently with a forward-advancing clock, the trailing window edge can misclassify. The same captured timestamp must drive both the server filter and the candidate predicate. Currently unspecified.

11. **[MEDIUM] Per-page complexity is unverified against the 10k-per-query cap.** The nested issues query pulls `state + assignee + labels(connection) + project + cycle + parent` inline (D1, §2.2). The `labels` sub-connection inside an issues connection multiplies complexity per row; with `first: 50` (edge-cases note) this is asserted "well under" the cap but never computed. An over-cap query fails outright (not a rate-limit) and the plan has no handling for a hard complexity rejection. Compute/bound it (and the labels sub-`first`).

12. **[MEDIUM] Clock-skew handling for epoch-ms resets is hand-waved.** D6 caps the wait at ~60s "to avoid an unbounded wait on a skewed clock," but a *backward*-skewed local clock makes `reset - now` negative → sleep 0 → immediate retry → `RATELIMITED` → retries burned. The cap only addresses forward skew. Use a minimum floor and the `X-RateLimit-*-Remaining` signal, not just the reset delta.

13. **[MEDIUM] The `archived` column is effectively dead / cannot be distinguished from a tombstone.** §3.1 models `archived INTEGER`, but Linear's `issues` connection excludes archived issues unless `includeArchived: true` is passed (not shown in any query). An archived issue therefore simply vanishes from the fetch and is tombstoned — it can never be observed to set `archived = 1`. Either fetch with `includeArchived` and reconcile the flag, or drop the column to avoid implying a state that never populates.

14. **[MEDIUM] Empty default team selection can poison the baseline.** `selected_team_ids_json` defaults to `'[]'` (§3.1) and panel "Todos" = selected teams. On first run before the user picks a team, a "full sync" over zero teams could set `baseline_done = 1` (§4.5 sets it "after the first full sync"). Then selecting teams later runs against an established baseline → notification storm for pre-assigned issues. Baseline must be gated on a non-empty selection actually having synced.

15. **[MEDIUM] Issue moved across teams is undocumented and untested.** The spec covers "deselecting a team tombstones its contents" but not an issue moving team A→B (both selected), or A→(unselected). The PK-upsert likely updates `team_id`, but the team-A tombstone-candidate pass and the viewer-assigned union interact in non-obvious ways (a moved-to-unselected-but-assigned issue survives with an unselected `team_id`, invisible in "Todos"). The ClickUp precedent had moved-task handling; this drops it. Add a scenario + test.

16. **[MEDIUM] Attachment/markdown links open arbitrary URL schemes in the OS browser without validation.** linear-task-panel: "clicking an attachment chip SHALL open the attachment's original URL in the browser." The URL is untrusted issue data; no http(s) scheme check is specified. `file://`, `javascript:`, or custom-scheme URLs handed to the OS opener are a real surface. Allow-list http/https before opening.

17. **[MEDIUM] Synthesized placeholder state/team rows can be tombstoned, causing oscillation.** §4.2 synthesizes placeholders for unknown `state_id`/`team_id`; §4.3 tombstones rows "absent from the fetch." A synthetic placeholder state is by definition never in Linear's fetch, so the next reconcile may tombstone it while issues still reference it. Placeholders need an exclusion from tombstoning (and a re-resolution path once the real row arrives).

18. **[HIGH] Test plan omits every one of the highest-risk paths.** §2.4/§4.8 cover the happy and simple-edge cases, but there is **no** test for: (a) truncated/rate-limit-interrupted pagination NOT tombstoning (#1); (b) sub-issue-before-parent ordering and out-of-scope parent (#2); (c) aged-out non-assigned eviction (#4); (d) account/viewer-switch baseline reset (#8); (e) inline-markdown image/script-scheme sanitization (#7); (f) exhausted-bucket reset selection (#6). Each maps to a critical/high behavior above; "the reconcile tests are the correctness core" (execution order #4) is undermined by these absences.

19. **[MEDIUM] Workspace-label (null `team_id`) reconcile authoritativeness is undefined.** §3.1 allows `team_id NULL` for workspace labels, and the per-team fetch is the authoritative scope. Workspace labels are not bound to any selected team, so the plan never says when a deleted workspace label is tombstoned, nor which fetch is authoritative over them. With `ON DELETE CASCADE` on the join, a wrongly-GC'd workspace label silently drops live issue↔label associations.

20. **[LOW] Lazy-comment refetch keyed on `updatedAt` advance is both over- and under-inclusive.** §4.6/D5: comments refetched when `updatedAt` advanced. A title/assignee edit bumps `updatedAt` and triggers a needless comment refetch (complexity cost), while the heuristic assumes every comment change bumps issue `updatedAt` — verify that holds, or comment-only changes are missed on the panel until detail-open.

The foundation, schema mapping, auth seam, and rate-limit-is-a-400 trap are handled competently. But #1 (data-destroying false tombstone on truncated fetch), #2 (FK-fatal parent ordering), and #3 (unverified `updatedAt`-on-reassign) are each individually disqualifying for a `critical` risk-tier change, and the test plan does not exercise them.

# Adversarial Review (Round 2) — linear-mirror

But the revision's central new mechanism — **set 3 (re-verify-currently-mine by id)**, which is what makes the critical #3 fix work — was bolted on without propagating its consequences into the query layer, the team-scope invariant, and the complexity budget. That reintroduces, in narrow form, the exact churn #9 eliminated, and it has no query defined to implement it. Combined with two scoping seams that survived into tasks.md, this is one more tightening pass short of clean.

2. **[MEDIUM→HIGH here] Set 3 over-fetches the entire assigned backlog every cycle, undercutting the complexity-budget rationale that justifies the whole bounded-poll design (D4).** Set 2 already returns every issue *currently* assigned to the viewer in the selected teams. The only issues set 3 actually needs are the **set difference** `mirror_mine − set2_returned − window_returned` (i.e. the issues the mirror thinks are mine that the cycle didn't otherwise surface — the un-assigned and moved ones). For a heavy user holding hundreds of issues, the plan doubles the per-cycle complexity on a 45s interval — the precise failure mode D4 exists to prevent. The plan's claim that set 3 is "bounded by how many issues you personally hold" is true but irrelevant: it's an unnecessary second full pass. Scope set 3 to the delta.

But tasks §1.3 defines `linear_set_key` and `linear_validate_key` as **separate** commands, and nothing forces a set to re-resolve the viewer. Acceptable for a read-only panel, but the assumption should be noted as a known limitation rather than left as an unverified premise.

The foundation, schema, reconcile gating, eviction, rate-limit handling, and security surface are now sound. The remaining defects are concentrated in one place: set 3 — the mechanism that fixes the critical #3 — was not carried through to the query layer (#1), the complexity budget (#2), or the team-scope invariant (#1/#4), and the viewer-reset trigger (#3) is too narrow. For a `critical` risk-tier migration these are load-bearing and individually implementable-wrong. One more tightening pass.

## New issues

1. **[HIGH] Per-cycle viewer-resolve wipes the mirror on a *failed* resolve, not just a changed one — data destruction on a transient error.** D5 and §4.1 say the poller resolves `viewer` every cycle and, "on mismatch/null `wipe_mirror()` + `baseline_done=0`." A `viewer` query can return `null`/error for reasons unrelated to an account swap: rate-limit (`RATELIMITED` on the very query, D6), network blip, transient keyring read (the spec's own "transient ≠ no-entry" scenario). If the resolved viewer is `None` because the *fetch failed* and the code follows the literal "mismatch/null → wipe," every transient viewer-query failure nukes the entire mirror and silently re-baselines. The reset MUST fire only on a **successfully resolved** viewer whose id differs from the stored one; a failed/empty resolve must skip the cycle (or skip the comparison), never wipe. The plan conflates "resolved null (failure)" with "stored null (post-key-change)" — exactly the ambiguity an implementer resolves the dangerous way. The round-2 fix traded a too-narrow trigger for a too-eager one. No test covers "viewer-resolve fails → mirror NOT wiped."

2. **[HIGH] Set-3 by-id *absence* semantics are undefined and the plan models a "404" that this query shape cannot produce — deleted-while-mine issues are never cleaned, and absence-based clearing would evict live issues.** D4's set-3 outcome handling enumerates three cases keyed on what the by-id fetch *returns* (`assignee != viewer` → clear; `team_id` unselected → evict; "404 → evict"). But `issues(filter:{ id:{ in:[…] } })` does **not** 404 for a missing id — a deleted, archived, or access-revoked issue is simply **absent** from the returned `nodes`. So:
   - The "deleted" case the plan thinks it handles via 404 actually arrives as *absence*, which the plan never handles → a deleted issue the mirror flags as the viewer's keeps `was_viewer_assigned=1`, is protected from eviction by that flag, and **lingers forever as "assigned to you."**
   - Conversely, if an implementer *does* act on absence ("delta ids not returned → gone → clear flag + evict"), then an **incomplete** set-3 batch (rate-limit / network give-up) marks the unreturned-but-still-mine issues as gone and **hard-deletes the user's real issues.**
   The correct rule — *"on a provably-complete set-3, a delta id absent from results → clear flag + evict; on an incomplete set-3, retain"* — is exactly what's missing, and it's the crux of the un-assignment/cleanup mechanism (the #3 fix). Define the absence rule and gate it on set-3 completeness. No test for by-id absence (deleted-while-mine) or for incomplete-set-3 non-eviction (§4.8 only tests the *returned*-as-unselected case).

3. **[MEDIUM→HIGH here] Set-3 `issues_by_id` has no pagination or id-batch chunking — a large delta is silently truncated, and the truncation isn't fed into the completeness gate.** Every other connection query carries `after`/`hasNextPage` handling; `issues_by_id(ids)` (tasks §2.2, §4.1) lists none. Two failure modes: (a) `id:{in:[…]}` over a large delta returns a connection capped at `first` (≤25 per D6) → only the first page of delta-mine is re-verified, the rest never have their assignee re-checked → un-assignment missed for them; (b) the `in` array itself may exceed a server-side bound. The delta is typically small but **unbounded** right after a bulk un-assignment — the exact moment correctness matters. And because set 3 is one of the "branches" the completeness gate watches (D4: "every paginated branch … reached `hasNextPage == false`"), an un-paginated set-3 has *undefined* completeness, which silently corrupts the `complete` flag that gates tombstoning. Specify chunking + cursor paging for the by-id query and its contribution to `complete`.

4. **[MEDIUM] Eviction is not gated on fetch completeness, and its safety now depends entirely on the (under-specified) set-3 flag maintenance.** D4 gates *tombstoning* on `complete` but states eviction only as "runs inside the same transaction, after upserts and tombstones" — never that eviction is skipped on an incomplete cycle. Eviction of a *genuinely* aged-out non-mine issue is safe regardless of completeness (it's out of scope by definition). But eviction's correctness for the viewer's own issues rests on `was_viewer_assigned` being accurate, which rests on set 3 (#2/#3). State explicitly that eviction's flag-clear inputs come only from a **complete** set-3, or eviction inherits the #2 data-loss path.

5. **[MEDIUM] Test plan omits every new set-3/viewer-resolve edge.** §4.8 adds tests for the *returned* cases (moved-to-unselected, delta-only fetch) but not for: (a) viewer-resolve **failure** → mirror NOT wiped (#1); (b) deleted-while-mine surfacing as by-id **absence** (#2); (c) **incomplete** set-3 → flags retained, no eviction (#2/#4); (d) delta larger than one by-id page → all re-verified (#3). These map 1:1 to the critical/high behaviors above; "the reconcile tests are the correctness core" is again undermined by the omissions.

6. **[LOW] `linear_set_key` clears `viewer_id` but the prior account's mirror rows remain visible until the next poll cycle (≤ interval).** D5/§4.5: the wipe happens on the *next cycle's* resolve. For up to `linear_poll_interval_secs` after a key swap the panel shows the old account's issues. Cosmetic, but for an account-switch this is a brief cross-account data exposure; consider an immediate wipe on `linear_set_key` (the clear is already there) rather than deferring to the cycle.

## Assessment

Rounds 1–2 hardened the schema, reconcile gating, rate-limit handling, and security surface, and the round-2 set-3/scoping defects are real fixes. But the same two mechanisms remain one tightening pass short: the per-cycle viewer-resolve can **wipe the mirror on a transient failure** (#1), and the set-3 by-id path **conflates absence with a non-existent 404, lacks pagination, and never defines the completeness-gated absence rule** (#2, #3) — leaving both a "stale-forever mine issue" leak and a "delete the user's real issues on an interrupted fetch" hazard. Both are data-destroying in the critical path and untested. Define (a) wipe only on a successfully-resolved differing viewer, (b) the complete-vs-incomplete set-3 absence rule, (c) by-id chunking/pagination + its completeness contribution, and add the four tests.

- **r3 #2** (set-3 absence semantics / phantom 404): D4 now keys outcomes on **presence**, with absent-on-complete → clear+evict and absent-on-incomplete → retain. Spec scenario + tests added. Fixed.
- **r3 #3** (by-id pagination): D4 + tasks §2.2 chunk ids ≤25/batch, cursor-paginate each, and feed `complete`. Test added. Fixed.

- **r3 #5** (tests): §4.8 adds viewer-resolve-failure, deleted-while-mine-absence, incomplete-set-3, oversized-delta. Fixed.
- **r3 #6** (deferred set_key wipe): §1.3/D5 now wipe immediately on `set_key`. Fixed *in trigger* — but see #1/#2 below for what it opened.

## New issues

1. **[HIGH] Account-swap race: an in-flight reconcile commits old-account data *after* `set_key` wipes, and the per-cycle backstop can no longer catch it because `set_key` nulled `viewer_id`.** The reconcile is atomic against *reads* (§4.2/r2 #6) but nothing guards `set_key`'s wipe against a concurrent poll's *write*. Sequence: cycle N starts, resolves the **old** viewer (old key still active), matches stored id, fetches old-account issues (slow network); user swaps the key → `set_key` runs `wipe_mirror()` + `viewer_id=null` + `baseline_done=0` (§1.3); cycle N then commits its old-account upserts, **re-populating the just-wiped mirror**. Cycle N+1 resolves the **new** viewer, finds stored `viewer_id == null` → D5's "stored-null is a *seed*, not a *mismatch*" → it **seeds without wiping**. The old-account rows survive: they belong to teams not in the new selection, so the completeness-gated tombstone candidate ("belongs to a selected team") never fires (D4), and if they were in-window they aren't aged-out so eviction skips them — **cross-account data lingers indefinitely**. The design needs an epoch/generation guard: capture a key-generation token at cycle start and, inside the reconcile transaction, abort the commit if the token changed (or have `set_key` cancel/await the in-flight cycle). The plan's only hook is the vague "poller restart hooks" (implementation.md, mod.rs), which does not imply aborting an already-fetched cycle's commit. No test for "key swapped mid-cycle → old data not committed."

2. **[MEDIUM→HIGH here] The per-cycle viewer backstop (the r2 #3 / r3-verified mechanism) is now unreachable — it can only ever seed, never detect a change.** D5 keeps the per-cycle resolve "as a backstop" and spec scenario "Baseline resets on viewer change" asserts the poller "finds a `viewer_id` different from a **non-null** stored one." But `set_key` is the **only** key-setter, and it now always nulls `viewer_id` before the next cycle. So the only states the per-cycle resolve can observe are *matches* (normal) or *null* (post-`set_key` seed) — the "non-null and differs" branch the scenario tests is **dead code** under the design's own invariants. This is not merely cosmetic: it is precisely *why* #1 isn't caught (the safety net the plan relies on to re-detect an account mismatch can never fire). Either (a) make the per-cycle resolve compare against a stored token that `set_key` does **not** clear (so a torn swap is detectable), or (b) drop the claim that the per-cycle resolve is a backstop and lean entirely on the epoch guard from #1. As written, the plan presents a load-bearing defense that its own state machine disables.

3. **[MEDIUM] Set-3 outcome enumeration omits the "returned, still the viewer's, still in a selected team" case, and the by-id query's relation depth is unspecified → undefined flag handling or label-stripping.** D4 enumerates exactly three set-3 outcomes: `assignee != viewer` → clear; `team_id` unselected → evict; absent → evict/retain. But the delta `mirror_mine − ids(set1) − ids(set2)` will contain a still-mine, selected-team issue **whenever set 2 was incomplete** (rate-limited) and missed it — that issue comes back from set 3 as `assignee == viewer`, team selected, and matches **none** of the three rules. The plan never says what happens (keep flag? refresh display? evict?). Tied to this: tasks §2.2 describes `issues_by_id` as the "delta re-verify… to learn their fate" without specifying whether it fetches the nested `labels`/`state`/`project` relations. If it is a lean assignee/team query and §3.2's per-issue label reconcile ("delete-absent + insert-present within the issue's upsert") runs on it, the upsert **strips the labels off the user's own backlog issue**. Specify (a) the fourth outcome explicitly (returned-still-mine-still-selected → retain flag, no evict), and (b) that set-3 upserts either fetch full relations or perform a *targeted* update (assignee/state/team only) that does **not** blanket-reconcile labels/comments.

4. **[MEDIUM] Team selection (`selected_team_ids_json`) is not reset on account swap.** `set_key` wipes the issue mirror and nulls `viewer_id`/`baseline_done`, but the plan never says it clears `selected_team_ids_json` (§1.3, §3.1). After swapping to a different org, the persisted team ids belong to the **old** org and match nothing in the new one → `issues_page(team_ids=selected)` and `viewer_assigned_issues(team_ids=selected)` return empty, `baseline_done` is correctly never set (gated, #14), but the user sees a permanently empty panel with no signal why until they manually re-pick teams. Either clear the selection on `set_key` (forcing a fresh pick) or validate stored team ids against the new viewer's teams on the first post-swap cycle and surface "select teams."

5. **[LOW] Spec scenario wording "regardless of whether the key was changed via set or validate" overstates `validate`.** The "Baseline resets on viewer change" scenario implies `linear_validate_key` participates in the reset, but `validate` does not persist a key (D2: "validation returns the resolved `viewer`, not the key") so it cannot change the active account, and tasks §1.3 only wires the wipe into `set_key`. Align the scenario text with the actual triggers (set_key immediate wipe + per-cycle resolve) to avoid an implementer wiring a needless wipe into `validate`.

## Assessment

Rounds 1–3 hardened the schema, reconcile gating, rate-limit handling, security surface, and the set-3/viewer-resolve mechanics, and the round-3 fixes are real. But r3 #6's immediate-`set_key`-wipe-plus-null-seed introduced an **account-swap race** (#1) that the per-cycle backstop **can no longer catch** because the same change made that backstop unreachable (#2) — a coupled, data-lingering hole in the critical path, untested. The set-3 outcome table is also missing its fourth (set-2-incomplete) case with an attendant label-strip ambiguity (#3), and team selection survives an account swap (#4). Add: (a) a key-generation/epoch guard so an in-flight cycle cannot commit across a `set_key`, (b) a backstop comparison that a torn swap can actually trip (or honest removal of the backstop claim), (c) the explicit returned-still-mine-still-selected outcome plus set-3 relation-depth/label-reconcile semantics, (d) team-selection reset on swap, and the corresponding tests.

# Adversarial Review (Round 5, final) — linear-mirror

Reviewing as skeptic. The round-4 fix that previous rounds kept reopening — the account-swap path — was finally re-architected around a **key-generation epoch** rather than a viewer-id comparison, which is the right structural move (it stops chasing the comparison-becomes-dead-code regress). I verify each round-4 finding below, then probe the epoch mechanism for a new hole. The critical/high paths are now sound; residual issues are low.

## Round-4 findings: verified fixed (not re-raised)

- **r4 #1** (account-swap commit race): D5 + new spec requirement *"Account swap is isolated by a key-generation epoch"* + tasks 4.1 now capture `key_generation` at cycle start and **re-read it inside the reconcile transaction, discarding the commit if it changed**. Scenario *"Key swapped mid-cycle does not re-populate the wiped mirror"* added. Because both `set_key`'s wipe and the reconcile are SQLite write transactions (serialized), the in-flight cycle either reads the bumped generation and rolls back, or commits-then-gets-wiped — old-account data cannot survive either ordering. Fixed.
- **r4 #2** (dead viewer backstop): D5 now **honestly removes** the backstop — "there is no 'non-null differs → wipe' branch (the epoch already handled the swap)." The cycle's viewer resolve only seeds the null slot; the swap defense is the epoch, not a comparison that could never fire. Fixed.
- **r4 #3** (set-3 fourth outcome + label strip): D4's outcome table now enumerates the *"Returned, `assignee == viewer`, team still selected → retain flag, full upsert, no evict"* case (set-2-incomplete), and tasks 2.2/4.3 specify the by-id query *"fetches the full nested issue shape… so the set-3 upsert reconciles labels correctly instead of stripping them."* Fixed.
- **r4 #4** (team-selection reset on swap): `linear_set_key` now also clears `selected_team_ids_json` (D5, spec, tasks 1.3), with the rationale (stale old-org ids → permanently empty panel) captured. Fixed.
- **r4 #5** (validate wording): the spec requirement states *"Validating a key SHALL NOT participate in the reset (it does not persist a key)"*; the old "via set or validate" scenario language is gone. Fixed.

## Residual issues (low — not disqualifying)

1. **[LOW] The epoch re-read is race-free only if the reconcile transaction holds the write lock from the start (`BEGIN IMMEDIATE`).** D5/tasks 4.1 say the cycle "re-reads `key_generation`… inside the reconcile transaction, aborts if changed." If implemented as a *deferred* transaction that reads the generation (shared lock) and only later upgrades to a write lock on the first upsert, a `set_key` wipe+bump could commit between the read and the upgrade, and the cycle would then see SQLITE_BUSY (handled) — but if the read snapshot is reused, it could observe the *stale* generation. The design intent is correct; the implementation must make the generation re-read the first statement of an IMMEDIATE (write-locked) transaction so the snapshot can't predate a concurrent `set_key`. Worth one line in tasks 4.1; not a plan-level defect.

2. **[LOW] Confirm notification emission is gated on the *committed* reconcile, not on the in-memory diff.** tasks 4.7 ties `linear:assigned` to "after each committed reconcile," which correctly suppresses a discarded (epoch-aborted) cycle's notifications — an in-flight old-account cycle that gets rolled back must not fire a stale `notify-send`. This is already implied by the wording; just ensure the assignment-notification side effect physically follows the commit (and the epoch check), not the pre-commit fetch diff. Covered by existing language; flagging so it isn't lost in implementation.

3. **[LOW, residual] Lazy-comment under-inclusiveness** (carried from r1 #20): comment-only changes that don't bump issue `updatedAt` surface only on detail-open. Explicitly recorded as a known limitation in D5; acceptable for a read-only panel.
