=== Round 1 (claude/skeptic) ===
I've verified the plan against the real `linear/` and `clickup/` sources. Findings below, with severity escalated per the adversarial mandate.

## Review — linear-writeback

1. **[HIGH] Standalone detail comment composer collides with the structural token boundary — an un-closed gap.** Task 6.1 adds a standalone "comment composer" as a detail write control (alongside the direct state/assignee controls), and the proposal walk treats "post a comment → see it" as a standalone action. But Decision 5 / Task 5.1 mandate that **all** comment writes "execute solely via `execute_closure(token,...)` — not directly-invokable per-write commands." The plan never specifies how the standalone composer reaches the API. This is a **new divergence from ClickUp**, where comments were *never* a standalone surface — `clickup/writeback.rs:19-24` states explicitly "`post_comment`/`verify_comment_landed` are NOT Tauri commands… Module 4 wraps them behind the confirmation-token gate." An implementer mirroring the adjacent direct state/assignee commands will reach for an obvious `linear_post_comment` direct command and breach precisely the boundary this change exists to defend (auto-postable, team-visible writes against a full-access personal key). The plan must state that the standalone composer routes through `request_closure_token(issue, None, comment)`/`execute_closure` (and the resulting confirm UX), or define a renamed comment-token command. Cite: Task 6.1, Task 5.1, Decision 5; `clickup/writeback.rs:19-24`.

2. **[HIGH] Mutation `success` field is never checked → silent no-op writes.** Linear's `issueUpdate`/`commentCreate` payloads carry `success: Boolean!`, and `execute<T>` only bails on a GraphQL `errors` array (`client.rs:122-130`) — it does **not** inspect `success`. A `{"data":{"issueUpdate":{"success":false}}}` with no `errors` is classified `Ok`, so the write command returns success: the registry records the write, the frontend overlay clears, but nothing changed upstream. The echo poll then sees `server == pre_write → Unrelated` (no warn, entry lingers to TTL), and the overlay reverts only on the next reconcile — a silent divergence the user believes succeeded. Task 1.1 and the test plan never require asserting `success == true` (or verifying the returned issue value). Cite: Task 1.1, `client.rs:122-130`.

3. **[HIGH] `read_team_states` / state-validation must exclude synthetic placeholder states.** `linear_workflow_states` carries a `synthetic` flag; `ensure_state_placeholder` (`mirror.rs:67-74`) inserts rows with `synthetic=1` and name `"(state {id})"` on paging skew. The plan's `read_team_states` ("ordered by `position`", Task 1.2 / implementation.md) and the server-side validation read this table with **no synthetic filter**. A placeholder id could be offered in the picker *and* pass validation → `issueUpdate` with a fake state id → upstream rejection or a wrong-state write. Both the picker read and the validation helper need `WHERE synthetic = 0`. Cite: `mirror.rs:67-74`, Task 1.2, Decision 5.

4. **[MEDIUM] "Echo dedup for state" is largely vacuous; the genuine new work — conflict emission — is under-specified.** The poller's *only* notification is `newly_assigned → linear:assigned` (`mod.rs:1033`, `:1111`), gated post-baseline; state changes emit no per-field event, just the generic `linear:changed` + a blind upsert (`poller.rs:146`, `:200`). So "echo dedup so our own state edit isn't reported as a remote change" suppresses nothing for state — there is no state-change notification to suppress. The real new machinery is a **per-issue conflict event**: ClickUp emits `clickup:write-conflict` (`clickup/writeback.rs:164-170`), but the Linear plan never names a `linear:write-conflict` event nor specifies that `reconcile` must now surface per-issue conflicts (it returns no such field today). Reframe Decision 2/3 and add the conflict-emit plumbing to the tasks. Cite: Decision 2/3, `poller.rs:262-265`, `mod.rs:1031-1035`.

5. **[MEDIUM] "Last-writer-wins by `updatedAt`" is a misnomer copied from ClickUp.** With optimism living only in the frontend overlay (Decision 1), the durable mirror never holds the local edit and the poller blind-upserts server truth (`poller.rs:146`/`:200`). There is no code path where the local value wins and no `updatedAt` comparison ever occurs — the resolution is "server always wins; warn iff it diverged from what we wrote." Decision 3 and the spec ("scalar conflict resolution… last-writer-wins by `updatedAt`") could push the implementer to add a pointless timestamp gate. Restate as server-wins + warn-on-divergence. Cite: Decision 3, spec §"Echo dedup and scalar conflict."

6. **[MEDIUM] "Echo before the set-diff inside reconcile" over-constrains location and fights `reconcile`'s purity.** `newly_assigned` is computed inside the pure, heavily-unit-tested `reconcile(conn, cycle, gen)` from the committed DB mine-set (`poller.rs:262-265`); the in-memory `WritebackRegistry` and the `AppHandle` are not in scope there, and you cannot `app.emit` a conflict inside the generation-guarded transaction. Forcing the echo logic "before `newly_assigned_ids`" inside `reconcile` either couples the DB-reconcile tests to writeback state or is unimplementable as written. The clean, equivalent point is the run loop between `reconcile` and `notify_assignments` (`mod.rs:1033`): filter `outcome.newly_assigned` against the registry before notifying. The real contract is "before the **notification**," not "before the set-diff." Cite: `poller.rs:159-265`, `mod.rs:1028-1035`, Task 3.1.

7. **[MEDIUM] TOCTOU between API-ack and `registry.record()`.** Routine writes (mirroring `clickup/closure.rs:311-318`) record *after* the `await` returns. A poll that fetches+reconciles in the window between the write landing on Linear and `record()` executing sees the new server value with no registry entry → spurious `linear:assigned` for an own assign-to-me (and a silent blind-upsert for state). This is the same class as the documented "benign crash-loss" but needs no crash. Either record a provisional entry before the call (clearing on failure) or at minimum document it alongside the crash-loss risk. Cite: Decision 2, Risks, `clickup/closure.rs:311`.

8. **[MEDIUM] Closure write order unspecified; local close-out precedes the Linear writes.** ClickUp deliberately does status-first → comment so the reversible half precedes the irreversible one (`clickup/closure.rs:24-32`); Task 5.3 / implementation item 6 pin no order. Worse, Task 5.3 applies the local close-out (unbind + marker) **before** the Linear writes, so a failed state write leaves the session already unbound — the "comment posted; state failed — retry state?" surface must carry the `issue_id` explicitly (the binding is gone, and for a ship-triggered closure the session keeps running). Specify state-first ordering and an issue-id-carrying retry. Cite: Task 5.3, `clickup/closure.rs:24-32`.

9. **[LOW] GraphQL field casing untested.** `IssueUpdateInput` fields are `stateId`/`assigneeId` and `commentCreate` input is `{issueId, body}` (camelCase); the Rust `IssueUpdateInput { state_id, assignee_id }` must serde-rename. The "builds the correct GraphQL shape" tests (Task 1.3) must assert camelCase keys (and select `success`, per #2). Cite: Task 1.1/1.3.

10. **[LOW] Ship hook must use the *shipped* session id, not the active session.** `ShipDialog.tsx:308-318` resolves the binding via `shippedSessionId` precisely because the active session may differ if the user switched tabs mid-ship; the Linear plan ("a session with an `active_linear_issue_id`", Task 6.4) drops that nuance. `resolveActiveLinearIssueById` must be called with the shipped session id. Cite: `ShipDialog.tsx:308-318`, Task 6.4.

11. **[LOW] assign-to-me with `viewer_id == None` would unassign.** `viewer_id` (`mirror.rs:331/336`) can be unresolved; an assign-to-me that falls through to `assigneeId: null` does the *opposite* of intent (clears the assignee). Guard with an explicit error. Cite: spec §"Assignee set-to-me," `mirror.rs:336`.

12. **[LOW] Spec scenario "Own write is not re-notified" is vacuous for state** (same root as #4). Tighten so it doesn't imply a state-notification suppression that has nothing to suppress.

**Verified-sound (no action):** Decision 1 (frontend-only optimism) genuinely closes the crash/split-brain hole and is correctly argued against the durable-optimism alternative; the token-store port is a faithful, single-use, TTL'd structural gate (`clickup/closure.rs`); the assignee set-to-me/clear scoping correctly reflects the mirror holding no member directory; the comment post-once + ambiguous-failure re-fetch model is carried correctly; the `KeyC` verb is genuinely free (`LinearTaskDetail.tsx:57` = S/W/P/B/R); push-only correctly does not offer (`ShipDialog.tsx:328` uses `git_push`, no offer).

#1, #2, and #3 are correctness/security holes that no task or test currently pins, and #3 is a path to writing an invalid state upstream.

VERDICT: REVISE

### NEW issues introduced by the revisions

**N1. [HIGH] Routing the standalone detail comment composer through `execute_closure` silently closes out the issue and unbinds the session.** The #1 fix (Decision 4, Task 6.1) routes the detail's "post a comment" composer through `request_closure_token(issue, state=None, comment=text)` → `execute_closure`. But Decision 6 / **Task 5.3** mandate that `execute_closure` performs the local close-out — **unbind (`linear_unbind_issue`) + the worked-closed marker — "applied *regardless* of the Linear outcomes" as "the core local act."** There is no parameter on the token or `execute_closure` distinguishing "post a routine comment" from "close this issue out." So a user posting a quick comment from the issue detail would **unbind the session from its issue and stamp it worked-and-closed** — a destructive side effect they never asked for, and for a still-running bound session this severs the binding that scopes future closure/ship offers. Decision 4 calls it "the comment-only token path," but Task 5.3 makes close-out unconditional on that path — an internal contradiction. This is a *new* surface with no ClickUp analog (round-1 #1 itself noted ClickUp never had a standalone comment surface). The plan must add a non-closing comment-token command (`request_comment_token`/`execute_comment`) or a `close_out: bool` on the token/`ClosureResult`, and Task 5.3 must gate the unbind+marker on it. Cite: Decision 4, Decision 6, Task 5.1/5.3/6.1.

**N2. [HIGH] The echo/conflict comparison is specified against data the run loop does not hold; the state-conflict half — "the real new machinery" — has no source at the named location.** Decision 2, implementation.md step 4, and Task 3.1 all place the value comparison "in the run loop, between `reconcile` and `notify_assignments` (`mod.rs:1033`), **which holds the registry, the AppHandle, and the fetched `cycle` (carrying each issue's current `state.id` and `assignee.id`)**." Verified false: the run loop calls `poller::run_cycle(&client, &db)` (`mod.rs:1028`) which **constructs `FetchedCycle` locally and drops it at return** (`poller.rs:441-453`); it returns only `ReconcileOutcome`, whose fields are `discarded/tombstoned/evicted/newly_assigned/baseline_set/fetch_error` (`poller.rs:47-60`) — **no per-issue server state/assignee values**. So at `mod.rs:1033` the cycle is gone and the outcome carries no field values.
- The **assignee-echo filter** *is* implementable there (registry + `outcome.newly_assigned` suffice — filter ids whose registry `Assignee` write == viewer). 
- The **state-conflict emit** — which Decision 2 calls *"the real new machinery this change adds"* — is **not** implementable as written: nothing at `mod.rs:1033` exposes the server's current state per issue. ClickUp solved the identical problem with a **poller struct holding `self.app` + the registry, comparing inline in the fetch loop** where the fetched `task` is in scope (`clickup/writeback.rs:check_echo` called from `clickup/poller.rs:290` via `task_field_value(task, …)`, emit at `:943`). Linear's architecture — a **pure free-function `reconcile(conn, cycle, gen)`** plus an outcome-returning `run_cycle` — does not map onto that, so the "near-1:1 port"/"mirror ClickUp" framing hides a genuine architectural gap. The plan must pick and specify one: (a) extend `run_cycle`/`ReconcileOutcome` to surface a per-issue server-value map, (b) re-read the just-committed mirror for each registry entry in the run loop (mirror = server truth post-reconcile — but that contradicts "compare against the fetched cycle"), or (c) thread registry+app into `run_cycle` and compare inline à la ClickUp (contradicts the plan's "NOT inside `run_cycle`"). As written, an implementer hits a dead end. Cite: Decision 2, impl step 4, Task 3.1; `mod.rs:1028-1036`, `poller.rs:441-460`, `poller.rs:47-60`, `clickup/poller.rs:290`.

**N3. [MEDIUM] `run_cycle`'s signature is `(client, db)` — neither the registry nor the AppHandle is in scope, and no task specifies threading them.** Every viable resolution of N2 requires changing `run_cycle`'s parameters and/or return type (and `manage()`-ing `WritebackRegistry` so the poller task can reach it — Task 2.2 registers it as `tauri::State`, but the poller loop at `mod.rs:1028` is a spawned task, not a command, so `tauri::State` injection doesn't apply there — it needs an `Arc` handle threaded explicitly). The plan treats the comparison as a drop-in at a seam that currently has access to none of its inputs. Cite: `poller.rs:344-347`, Task 2.2, Task 3.1.

**N4. [LOW] The assignee "don't-touch vs set-null" distinction depends on an unstated `skip_serializing_if`, and the don't-touch case is untested.** Task 1.1 uses `assignee_id: Option<Option<String>>` to distinguish omit/null/set, but for `None → omit the key` the struct field needs `#[serde(skip_serializing_if = "Option::is_none")]`; without it, serde emits `assigneeId: null`, turning a state-only write into an unintended unassign. Task 1.3 tests state-only `{stateId}` and assignee-null, but does not explicitly pin that a state-only `IssueUpdateInput` **omits** `assigneeId` (the failure mode). Add the attribute and an explicit "assigneeId absent" assertion. Cite: Task 1.1/1.3.

**N5. [LOW] The #11 viewer-None guard is frontend-only by the plan's own wording; the backend cannot enforce it.** Task 1.4 states the None-guard "is the UI's responsibility" and `linear_set_assignee(issue_id, assignee_id: Option<String>)` legitimately accepts `None` for unassign — so the backend command cannot distinguish "unassign" from "assign-to-me with an unresolved viewer." Decision 8 says assign-to-me "SHALL error when viewer_id is None," but that guard lives only in the renderer, which is the exact trust boundary this change elsewhere refuses to rely on (Decision 5). Since assignee writes are intentionally not token-gated (reversible carve-out) this is an accepted-risk-consistent gap, not a contradiction — but it should be named as frontend-only, not implied structural. Cite: Decision 8, Task 1.4; confirmed `SyncState.viewer_id: Option<String>`, `mirror.rs:331`.

## Round 3 — verification of N1-N5 resolutions

**N1 (comment-token closes-out) — RESOLVED, coherent.** The token now carries a `close_out: bool` discriminant with two distinct request commands: `request_comment_token(issue, comment)` (`close_out=false`, `state=None`) and `request_closure_token(issue, state?, comment?)` (`close_out=true`), both feeding a single `execute_gated_write`. Task 5.3 gates step (3) unbind+marker on `close_out=true`; the comment-only path skips it. Pinned by a test (Task 5.4: "a `close_out=false` comment token posts the comment but does NOT unbind or write the closed-out marker") and a spec scenario ("Posting a comment does not close the issue out"). The structural collision is gone — posting a comment can no longer sever the binding. Security boundary intact (both paths still token-gated). No new defect: `request_comment_token` has no `state` param, so a comment token structurally cannot carry a state write.

**N2 (run-loop data source dead-end) — RESOLVED, implementable.** Verified the premise was real: `run_cycle(client, conn_lock) -> Result<Option<ReconcileOutcome>>` (poller.rs:344) builds `FetchedCycle` locally and drops it; `ReconcileOutcome` (poller.rs:47) carries no per-issue field values. The plan now picks **option (b): re-read the post-reconcile mirror.** This is sound because `reconcile` blind-upserts the fetched payload (`upsert_issue_full`, poller.rs) → the mirror IS server truth post-commit. At the notify point (mod.rs:1028-1033) `db` and `app` are both in scope, so re-locking to read `state_id`/`assignee_id` per registry entry is feasible. The echo filter mutates `outcome.newly_assigned` before the `pre_baseline && !outcome.newly_assigned.is_empty()` notify guard. The out-of-window edge (evicted/stale mirror row → skip) is correctly reasoned (no fresh remote value to conflict). The false "compare against the fetched cycle" framing was removed consistently across design.md §Decision 2, implementation.md step 4, Task 3.1, and spec.md.

**N3 (registry reach in spawned task) — RESOLVED, grounded.** Confirmed `app.state::<WritebackRegistry>()` is exactly the pattern ClickUp's spawned poller uses (`clickup/poller.rs:1205`), and `.manage()` registration is the established idiom (lib.rs:293-294). The plan correctly distinguishes this from command-only `tauri::State` injection. Task 2.2/3.1/6.5 are consistent.

**N4 (serde skip) — RESOLVED.** Task 1.1 now mandates `#[serde(skip_serializing_if = "Option::is_none")]` on the outer assignee `Option`; Task 1.3 adds the explicit "state-only `issueUpdate` **omits** `assigneeId` entirely — not `null`" assertion. Decision 8 carries the rationale.

**N5 (viewer-None guard) — RESOLVED as requested.** Decision 8 + Task 1.4 name it "frontend-only by design," explain why the backend cannot distinguish unassign from assign-to-me-with-unresolved-viewer, and tie it to the reversible-write carve-out. N5 was a LOW ask to *name* it, not re-architect it — done.

**No new defects introduced.** I checked the adversarial seams: assignee conflict (remote reassign → `server ≠ written ≠ pre` → conflict, and `mine_now` excludes it so no spurious notify), assignee echo (`server == viewer` → filtered from `newly_assigned`), unassign echo, and the comment-only `execute_gated_write` order (state-skip → comment → close-out-skip). `viewer_assigned_issues` (client.rs:280) confirms the "in-window by construction" edge claim. The 12 round-1 issues remain closed. All five artifacts (proposal/design/implementation/tasks/spec) are internally consistent.
