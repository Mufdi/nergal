## 2. Optimistic overlay + recent-writes registry (no durable optimistic mutation)

- [x] 2.1 Optimism lives in a **frontend overlay atom** (`linearOverlayAtom` in `stores/linear.ts`, mirroring `clickupOverlayAtom`) keyed by issue+field; the durable mirror is updated only by the reconcile after an API ack (Decision 1). On API failure the overlay reverts + toast. The mirror is never written a value that was not server-acked.
- [x] 2.2 `linear/writeback.rs`: `WritebackRegistry` — record `(issue_id, field, written_value, pre_write_value, at)` for echo/conflict comparison. **Record a provisional entry BEFORE the API call and clear it on API failure** (review #7 — closes the TOCTOU where a poll lands between the write hitting Linear and the command resuming). Fields = `State`, `Assignee` (both scalar). Entries TTL out at ≥ ~2× the poll interval (derived from cadence). Register as `tauri::State` in `lib.rs`.
- [x] 2.3 Tests: API failure clears the provisional record AND reverts the overlay AND never persists to the mirror; recent-writes TTL ≥ 2× poll; expired entries purged.

## 3. Echo dedup + conflict resolution (value comparison)

- [x] 3.1 **In the run loop, after `run_cycle` returns and before `notify_assignments` (`mod.rs:1033`) — NOT inside the pure `reconcile`/`run_cycle`** (reviews #6, N2: `run_cycle(client, db)` drops the `FetchedCycle` and returns only `ReconcileOutcome` with no per-issue values). Data source = the **post-reconcile mirror** (server truth, since `reconcile` blind-upserts the fetched payload): for each `recent_writes` entry read the issue's current `state_id`/`assignee_id` from the mirror → `check_echo` against written/pre values → own echo (clear entry, suppress) / unrelated (server == pre-write) / scalar conflict. **Filter own-echo issues out of `outcome.newly_assigned` before `notify_assignments`** so an own "assign to me" never self-fires `linear:assigned`. The `WritebackRegistry` is reached from the spawned poller task via `app.state::<WritebackRegistry>()` (review N3 — NOT `tauri::State` injection, which is command-only); if an issue we wrote fell out of window (no mirror row), skip it (no fresh remote value to conflict).
- [x] 3.2 Conflict (Decision 3, scalar-only, server-wins): the blind upsert already lands server truth; on a divergence (server value ≠ written ≠ pre-write) **emit a NEW `linear:write-conflict` event** (`{issue_id, field, your_value, remote_value}`, mirror `clickup:write-conflict` shape `writeback.rs:164`). NOT "LWW by `updatedAt`" (review #5 — no timestamp gate, no local-wins path exists). No additive-merge branch (Linear has no in-scope set field — documented divergence from ClickUp).
- [x] 3.3 Tests: own state write produces no `linear:write-conflict` (value match); **regression test: own assignment-write is filtered from `newly_assigned` before notify (own assign-to-me does not self-notify)**; divergent remote state → `linear:write-conflict` emitted; generation-guard discards a mid-cycle account swap (reconcile purity preserved — the echo/conflict layer reads the registry, never mutates `reconcile`).

## 5. Structural write boundary + write-back-on-done closure

- [x] 5.1 `linear/closure.rs`: confirmation-token store (`Mutex<HashMap<token, GatedToken>>`, 30s TTL, destructive single-use `take`). Token carries a **`close_out: bool`** discriminant (review N1). TWO request commands: `request_comment_token(issue_id, comment)` (issues `close_out=false`, `state=None` — the detail composer's path) and `request_closure_token(issue_id, state_id?, comment?)` (issues `close_out=true` — the closure prompt). Both validate the state against the issue's non-synthetic team states (when present) + sanitize the comment at issuance + scope the token to its tuple. The **closure and comment** writes execute solely via a single `execute_gated_write(token, ...)` — not directly-invokable per-write commands. A renderer cannot auto-fire them.
- [x] 5.2 `026_linear_closed_out.sql` (`issue_id TEXT PRIMARY KEY, closed_at INTEGER NOT NULL`) + `mirror::mark_closed_out`/`read_closed_out` + commands `linear_mark_closed_out`/`linear_read_closed_out` (mirror `019_clickup_closed_out.sql` + `clickup/mod.rs:337/351`). Register migration in the migrations list.
- [x] 5.3 `execute_gated_write` on confirm (token consumed), in this order (reviews #8, N1): (1) **state** write first (reversible/retryable — mirror `clickup/closure.rs:24`), (2) **comment** post second (irreversible), (3) the local close-out — unbind (`linear_unbind_issue`) + worked-closed marker — **only when the token's `close_out=true`**, applied **regardless** of the Linear outcomes (the core local act of a closure). A `close_out=false` comment-only token (detail composer) posts the comment and does NOT unbind or mark closed (review N1 — posting a comment must not close the issue). The state/comment halves are independent and optional (state-only is the common closure case; an empty comment posts nothing; neither selected + `close_out=true` → still closes out locally with no Linear write). Report per-write outcome; the comment is never optimistically rolled back. If the state fails but the comment posts, surface "state change failed — retry state?" **carrying the `issue_id` explicitly** (the binding is already unbound, and a ship-triggered session keeps running — the retry cannot re-derive the issue from the binding). `GatedWriteResult { state: StateOutcome, comment: CommentOutcome, closed_out: bool }`.
- [x] 5.4 Tests (Rust — the offer raise is frontend-only): no write without a valid token; token is single-use, short-lived, and scoped to its `(issue, state?, comment?, close_out)`; partial-failure result reports the irreversible half correctly; **a `close_out=false` comment token posts the comment but does NOT unbind or write the closed-out marker** (review N1); a `close_out=true` token with neither half still unbinds + marks closed without a Linear write.
